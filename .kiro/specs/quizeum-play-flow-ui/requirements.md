# 要件定義書: quizeum-play-flow-ui

## はじめに
本ドキュメントは、クイズ投稿SNS「quizeum」におけるホーム画面、クイズ探索、クイズ詳細、プレイ画面（通常・ウミガメのスープ含む）、結果画面、弱点克服、およびリーダーボードを含む、ユーザーのクイズプレイに関するフロントエンドUI要件を定義します。

**Phase 5（2026-06）**: クイズ詳細のリーダーボード表示を、初回プレイ／リプレイの二系統に改定し、正解数優先・同点時タイムの表示規則に合わせる（データ更新は `quizeum-core` が担当）。

**Phase 6（2026-06）**: ジャンルマスタ駆動の探索 UI。アイコンクリックは `/genres/[id]` 遷移に統一。複合検索はサジェスト付きジャンル入力・フィルタ変更時の `searchQuizzes`（デバウンス）・認証ユーザーのプレイ状況絞り込み（要件 1.3 完遂）を含む。ハードコード `GENRES` 廃止、ジャンル／タグ一覧にソートタブ追加。

**Phase 8（2026-06）**: ブックマーク画面をクイズ・リスト・問題の3タブに拡張し、プレイ／結果画面からの問題ブックマーク、および問題リスト（`listType: 'question'`）の連続プレイ導線を追加する（データ取得・永続化は `quizeum-core` が担当）。
**Phase 9（2026-06）**: トップページレイアウトのスタイリッシュ化、単一の統合検索バーによるクイズ名・作者・タグ・ジャンルのハイブリッド検索、クイックサーチチップ、検索中スケルトンプレースホルダーの導入。

**Phase 10（2026-06）**: ホーム統合検索のタグチップ化（スペース確定）とタグ・ジャンル名サジェスト、クイズ一覧カードの難易度を数値併記の星表記（例: ★ 7）へ変更しジャンル・出題形式を表示、ジャンル／タグ一覧ページのカード表示をホームと統一する（タグマスタ読み取りは `quizeum-core` が担当）。
2026-06-06 追記: 各検索フィールドのフォーカス時スマートサジェストを追加。ジャンル検索フィールドでは「直近検索ジャンル最大3件（`localStorage`）+ 週間プレイ数Top5」、統合検索フィールドでは「直近検索ワード（`localStorage`）+ 週間人気ワード／タグ各Top5」を空クエリ時に表示する（集計 API は `quizeum-core` が提供。ユーザー個人履歴は `localStorage` のみ）。

**Phase 11（2026-06）**: ホーム検索バー直下のジャンル／出題形式カードカルーセルによる**ホーム内フィルタ型**探索、既存ジャンルピルナビの置換、ジャンル別一覧ページへの検索バー・フィルタ追加（出題形式フィルタは `quizeum-core` が担当）。※ホームのカルーセル配置は Phase 21 で検索セクション内常時表示へ改定。

**Phase 12（2026-06）**: 主要画面（詳細、結果、ホーム、探索等）の Server Component 化、静的フレーム先行表示、および Suspense と Streaming による非同期スケルトン表示への移行。

**Phase 12 追補（2026-06-07）**: 本番プレイ（`/quiz/[id]/play`）およびテストプレイ（`/quiz/test-play/play`）の静的フレーム即時表示、`PlaySkeleton` + Suspense 適用。プレイ進行・セッション復元ロジックは変更しない。

**Phase 13（2026-06）**: 難易度の5段階化（1〜10から1〜5への変更）に伴い、クイズカード、詳細画面、および結果画面（投票含む）の星ゲージ表示と投票選択肢を1〜5の5段階へ変更します。

**Phase 14（2026-06-08）**: クイズ結果画面（本番・テストプレイ）において、各問題の回答と解説をアコーディオン化（初期は閉じた状態）し、体感難易度投票をクリック可能な★アイコン形式に変更する（投票値の永続化は `quizeum-core` の既存仕様を利用）。

**Phase 15（2026-06-08）**: 通常モード（`mode=normal`）のプレイフローを、回答後の即時正誤表示・「次へ」／最終問の「結果を見る」・わからないときのスキップ（不正解扱い）・楽観的結果画面遷移（「解答データを送信中...」廃止）に改定する。模擬試験・フラッシュカード・ウミガメのスープ・問題リストプレイは対象外。

**Phase 16（2026-06-09）**: 早押し形式（`quick-press`）の通常モードプレイにおける、区間累計の経過時間計測・問読み修了後の制限時間カウントダウン開始・不正解時の正解非表示・問題カードの初期レイアウト拡大を改定する。早押しタイム（押下〜回答秒数）の既存仕様は維持する。

**Phase 19（2026-06-09）**: クイズ詳細画面のプレイモード選択において、模擬試験・フラッシュカードがクイズ単位リーダーボード（初回プレイ・リプレイ）の対象外であること、および先にこれらのモードでプレイした場合は以降の通常モードプレイも初回プレイランキングに掲載されない旨の警告を表示する（LB 登録ルールは `quizeum-core` が担当）。

**Phase 20（2026-06-09）**: 〇×問題（`true-false`）のプレイ時に、大きな 〇／× ボタンを1タップするだけで即回答できる専用 UI を提供します。通常モードでは要件 17 のフィードバックフローと統合します。形式ラベル・探索カルーセルへの「〇×式」追加を含みます（形式解決・検証は `quizeum-core`、作問 UI は `quizeum-creator-dash-ui` が担当）。

**Phase 21（2026-06-09）**: ホーム画面のフィルタ UI を再編します。ジャンル・出題形式の横スクロールカルーセルを検索セクション内に移設し、アコーディオンによる折りたたみを廃止して常時表示とします。クイズ一覧は初回少量取得＋スクロール末端での自動追加読み込み（無限スクロール）に改修し、スクロール時は検索バー行を画面上部に固定表示します（段階的取得 API は `quizeum-core` が担当）。

## 要件

### 要件 1: ホーム画面 (Page: `/`)
**目的:** Quizeumのユーザーとして、タブ・ジャンル・複合検索でクイズを探索したい。それにより興味のあるクイズを見つけやすくなる。

#### 受け入れ基準
1. **ファーストビューの最適化**:
   - The Play Flow UI は、巨大なテキストバナーを廃止または縮小し、そのスペースをクイズコンテンツ表示に割り当てること。
   - The Play Flow UI は、検索バーを画面最上部の目立つ位置に優先して配置し、カテゴリーナビゲーションよりも視覚的優先度を高くすること。
2. **カテゴリー（ジャンル）表示の整理**:
   - The Play Flow UI は、`listActiveGenres` 由来のジャンル一覧について、1行に収まるスタイリッシュな「ピル」形式で横スクロール可能に表示し、縦方向の占有スペースを最小化すること。
   - The Play Flow UI は、主要なカテゴリー以外を「すべて見る」ボタン内に隠し、初期UIをクリーンに保つこと。
   - When ジャンルピルをクリックしたとき、the Play Flow UI は、ホーム内フィルタではなく `/genres/[genreId]` へ遷移すること（UI専用オプションはローカル定義可）。
3. **コンテンツ優先のレイアウト**:
   - The Play Flow UI は、画面の主要部分にグリッド形式で多数のクイズカードを表示し、1画面で発見できるクイズの数を最大化すること。
   - ホーム画面は「新着順」「人気順」「トレンド順」「タイムライン」（認証時のみ）のタブパネルを維持し、視認性の高いスタイルで切り替えをサポートすること。
4. **検索機能および UI/UX**:
   - **統合検索入力**: ジャンル、タグ、クイズ名、作者名を単一の検索バーで同時に検索可能とすること。
   - **検索バーの UI/UX**:
     - When ユーザーが検索バーを表示したとき、the Play Flow UI は、検索バーの左側に検索（虫眼鏡）アイコンを表示すること。
     - When ユーザーが文字を入力したとき、the Play Flow UI は、検索バーの右側にワンクリックで入力を全クリアできる消去（X）ボタンを表示すること。
     - When ユーザーが検索バーをフォーカスしたとき、the Play Flow UI は、ネオン調のグラデーション光彩のアウトラインをスムーズなトランジションで表示すること。
   - **クイックサーチチップ**:
     - When ユーザーが検索バーの直下に配置された人気のタグ（#ウミガメのスープ、#難問など）やおすすめジャンルのネオンバッジチップをクリックしたとき、the Play Flow UI は、そのテキストを検索バーに入力し、即座に統合検索を実行すること。
   - **スケルトンスクリーン**:
     - While 検索クエリの実行中（読み込み中）である間、the Play Flow UI は、クイズカードのプレースホルダーとなるスケルトンスクリーン（骨組みアニメーション）を表示すること。
   - **難易度・問題数・プレイ状況フィルタ**: 従来どおり、展開可能なフィルターパネル（SlidersHorizontal トグル）内で詳細指定ができること。ジャンル入力サジェストも含む。
   - **検索の実行**:
     - When 検索キーワードまたはフィルター条件が変更されたとき、the Play Flow UI は、デバウンス（300ms）を適用して `searchQuizzes` を呼び出し、条件を適用した結果をグリッドに反映すること。
5. **クイズカードの魅力向上**:
   - The Play Flow UI は、クイズカードにおいて、単一の固定電球アイコン表示を廃止し、クイズの内容を表現する固有のサムネイル画像を表示すること（サムネイルがない場合は適切なフォールバック表示を行う）。
   - The Play Flow UI は、各クイズカード内にタイトル、作成者、難易度、評価（スター数）などの情報を、整理された読みやすいレイアウトで表示すること。
   - The Play Flow UI は、各クイズカード内に明確な「プレイ」または「挑戦する」ボタンを設置し、ユーザーの直感的なインタラクションを誘導すること。
   - When 認証済みユーザーがカード内のブックマークボタンを押したとき、the Play Flow UI は、ブックマーク数を更新すること。未認証ゲストが押した場合は `/login` へ遷移すること。

### 要件 2: クイズ詳細画面 (Page: `/quiz/[id]`)
**目的:** Quizeumのユーザーとして、クイズのメタ情報・評価・ランキングを確認しプレイモードを選びたい。それによりプレイ前に内容を把握できる。

#### 受け入れ基準
1. クイズ詳細画面は、タイトル、説明、サムネイル、タグ、ブックマークトグルを表示すること。
1a. クイズ詳細画面に表示するジャンルは、ジャンルIDではなくジャンルマスタから解決した日本語表示名（例: 「プログラミング」等）とすること。また、その表示名の直前に小さなジャンルアイコン画像（iconImageUrl）をインラインで表示すること（アイコンがない場合はデフォルトのフォールバック表示を伴う）。
1b. クイズ詳細画面に表示する難易度は、`★★★☆☆`（★と☆の組み合わせ）のような5段階の等幅星ゲージで表現すること。
1b-1. The Play Flow UI shall 難易度に応じて星ゲージ（★）の色を変更し、難易度1〜2は緑、難易度3〜4は黄色からオレンジ、難易度5は赤へと変化するカラーグラデーション（または個別グラデーションカラー）で表現すること。
1c. クイズ詳細画面のバッジ・メタ情報エリアに、クイズ全体の出題形式（アイコンおよび日本語ラベル、例: `形式: 🌀 複合形式`）をバッジとして表示すること。
2. クイズ詳細画面は、`reviewBadge` と `reviewScore` を表示すること（7日間の再評価リセット期間中はマスク表示）。
3. クイズ詳細画面は、「通常モード」「模擬試験モード」「フラッシュカードモード」の3モードを選べるプレイパネルを表示すること。
3a. When [ユーザーが3モード選択可能なプレイパネルを表示したとき], the [Play Flow UI] shall [模擬試験モードおよびフラッシュカードモードがクイズ単位リーダーボード（初回プレイ・リプレイ）の対象外である旨、および先にこれらのモードでプレイした場合は以降に通常モードでプレイしても初回プレイランキングに掲載されない旨の警告文を、プレイパネル内の見やすい位置に表示すること（Phase 19）]。
4. プレイパネルでプレイ開始を押したとき、選択モードのクイズプレイ画面（`/quiz/[id]/play`）へ遷移すること。
5. ログイン中ユーザーが当該クイズの作成者である場合、「クイズを編集」ボタンを表示すること。
6. 「クイズを編集」を押したとき、クイズ編集画面（`/quiz/[id]/edit`）へ遷移すること。

### 要件 3: クイズプレイ画面 (Page: `/quiz/[id]/play`)
**目的:** クイズプレイヤーとして、カウントダウン・ヒント・セッション保護付きで解答したい。それにより公平で信頼できるプレイ体験を得られる。

#### 受け入れ基準
1. 通常または模擬試験モードで問題ごとに制限時間がある場合、カウントダウンを表示し、0秒で不正解として自動提出すること。ただし、通常モードの早押し形式（`quick-press`）問題については、制限時間の開始タイミングおよび経過時間の計測規則は要件 18 に従うこと。
1a. Where [通常モードで現在の問題が早押し形式（`quick-press`）である場合], the [Play Flow UI] shall [問読みが修了するまで、当該問題の制限時間カウントダウンを開始してはならない]。
2. 模擬試験モードでは問題ごとの制限ではなくクイズ全体の制限時間を適用し、前の問題へ戻って見直せること。
3. 「ヒントを表示」を押したとき、対応するヒント文を表示すること。
4. プレイ中、解答状態を `localStorage` に継続保存し、リロードやクラッシュから復元できること。
5. プレイ中にオフラインになった場合、オフラインで完了でき、完了時にオフライン結果画面を表示すること。
6. Where プレイモードが模擬試験・フラッシュカード・ウミガメのスープ・問題リストプレイである場合、the Play Flow UI shall 従来どおり、全問完了時に中間待機画面を挟まず結果の保存処理を実行し、結果画面（`/quiz/[id]/result`）へ遷移すること（Phase 15 以前の挙動を維持）。
6a. Where プレイモードが通常モード（`mode=normal`）である場合、the Play Flow UI shall 要件 17 に定義するフィードバックフローおよび楽観的結果遷移を適用し、本項 6 の自動遷移挙動に従ってはならない。

### 要件 4: 水平思考クイズ（ウミガメのスープ）プレイモード
**目的:** クイズプレイヤーとして、2カラムのAIチャットで水平思考クイズをプレイしたい。それにより質問しながら真相に辿り着ける。

#### 受け入れ基準
1. `lateral-thinking` 形式のとき、左にAIチャット、右にスクロール可能なQ&A履歴リストの2カラムレイアウトに切り替えること。
2. 未認証の場合、水平思考プレイ画面読み込み前に認証画面（`/login`）へ遷移すること。
3. 質問送信後AI応答待ちの間、AIチャット下部に灰色で「・・・AIが質問を分析中です」を表示すること。
4. 履歴と完全一致する質問を再送したとき、即時にキャッシュ回答を「📋 既存の回答」バッジ付きで返すこと。
5. 無料ユーザーが当該クイズの1日20回制限に達したとき、入力を無効化し「本日の残り質問数: 0/20」を表示すること。
6. 「真相を解き明かす」で真相要約を送信しAPIが合格判定したとき、クリア演出のうえ結果画面へ遷移すること。

### 要件 5: クイズ結果画面 (Page: `/quiz/[id]/result`)
**目的:** クイズプレイヤーとして、スコアと解説を確認し品質フィードバックを送りたい。それにより学習しコンテンツを評価できる。

#### 受け入れ基準
1. 結果画面は正解数／総問題数、経過時間、正誤一覧、各問題のマークダウン解説を表示すること。
1a. When [ユーザーが通常モードの最終問題で「結果を見る」を押して結果画面へ遷移したとき], the [Play Flow UI] shall [結果画面の静的フレーム（戻るボタン、コンテナ等）を即座に表示し、結果データの読み込み中は `ResultSkeleton` を表示すること。プレイ画面側で「解答データを送信中...」等の中間待機画面を表示してはならない]。
1b. While [通常モード完了直後に結果画面へ遷移し、サーバー上の attempt データがまだ利用できない間], the [Play Flow UI] shall [クライアント側に保持した完了データ（楽観的表示）を用いてスコアサマリーを表示し、バックグラウンドでの永続化完了後にサーバー attempt と整合させること]。
2. 結果画面は👍/👎の二値投票と体感難易度（1〜5）投票UIを表示すること。
2a. The Play Flow UI shall 結果画面の難易度表示や体感難易度投票における星（★）のUIについて、要件2.1b-1と同様に難易度に応じた緑→黄→赤のグラデーションカラーを適用すること。
3. 「問題の間違い指摘」でカテゴリ（誤字、事実、別解）と詳細を入力できるフォームを開けること。
3a. When クイズ全体に対する指摘（報告）を行うとき、the Play Flow UI shall 指摘カテゴリから「別解の追加」の選択肢を非表示にする、または選択不可能に制限すること（問題個別に対する指摘のときのみ「別解の追加」を選択可能とすること）。
4. 結果画面におけるクイズ作成者（作者）のフォロー機能：
   - Where ログイン中のユーザー（`user.id`）がクイズ作成者（`quiz.authorId`）と異なる場合、the Play Flow UI shall 作成者をフォローまたはフォロー解除できるボタンを表示すること。
   - When 結果画面がロードされたとき、the Play Flow UI shall ログインユーザーが作成者をフォロー中であるかどうかの状態を取得し、トグル形式（フォローする / フォロー解除）で初期表示を決定すること。
   - When ユーザーがフォローボタンをクリックしたとき、the Play Flow UI shall フォロー状態に応じて `followUser` または `unfollowUser` を実行し、フォロー状態の表示を更新すること。
   - Where ログイン中のユーザーがクイズ作成者と同一である場合、the Play Flow UI shall フォローボタンを表示しない（非表示）とすること。
5. オフライン時の制御：
   - While ネットワーク接続がオフラインである間、the Play Flow UI shall 良問評価投票、難易度投票、指摘フォーム送信、および作者フォローボタンを無効化（非活性）し、オフライン通知を表示すること。
6. Where 出題形式が連想クイズ（association）またはウミガメのスープ（lateral-thinking）である場合、the Play Flow UI shall 結果画面において、解答中に表示したヒントの一覧（またはヒント表示回数）およびウミガメのスープにおける質問回数を結果詳細情報として表示すること。
7. The Play Flow UI shall 結果画面に「もう一度プレイする」ボタンを表示すること。
7a. When ユーザーが「もう一度プレイする」ボタンを押したとき、the Play Flow UI shall 対象クイズの詳細画面（`/quiz/[id]`）に遷移すること。
8. The Play Flow UI shall 結果画面にクイズ作成者（作者）のユーザープロフィールページへのリンクを表示すること。
9. The Play Flow UI shall 同じ作者が作成した他の公開クイズを「おすすめクイズ」としてカード一覧またはリスト形式で表示すること。
10. The Play Flow UI shall 結果画面の「お疲れ様でした」カード領域に、クイズ全体に対するブックマーク（お気に入り）の登録・解除ボタンを表示すること。
11. The Play Flow UI shall 結果画面の指摘ボタンの横に「通報（報告）ボタン」を表示すること。
11a. When ユーザーが通報ボタンを押したとき、the Play Flow UI shall 通報理由等を入力・送信できる通報フォーム（またはモーダル）を表示すること。

### 要件 6: 弱点克服プレイ画面 (Page: `/quiz/review`)

3. 復習完了時、間違いリストを更新し `users.totalFailedQuestionsCount` を減算すること（`quizeum-core` 連携）。

2. タグ一覧（`/tags/[tagName]`）・ジャンル一覧（`/genres/[genreName]`）は該当クイズを人気順または新着順で一覧表示すること。

**目的:** 認証ユーザーとして、編集画面へのアクセスを適切に制限したい。それにより作成者以外が編集できないようにする。

#### 受け入れ基準

### 要件 9: クイズ単位リーダーボード表示（Phase 5）
**目的:** Quizeumのユーザーとして、初回プレイとリプレイのランキングを分けて確認したい。それにより初見と再挑戦の記録を誤解なく比較できる。

#### 受け入れ基準
1. クイズ詳細画面は、「初回プレイランキング（上位5名）」と「リプレイランキング（上位5名）」を**別セクションまたはタブ**で表示すること（総合リーダーボード画面とは別UI）。
2. 各ランキング表は、順位・ユーザー表示名・**正解数**・**合計解答時間（秒）**・達成日の列を含むこと。
3. 表示順は、正解数の多い順を第一キー、同点時は合計解答時間の短い順を第二キーとすること（`quizeum-core` の順位規則と一致）。
4. 各セクションに記録がない場合、「まだ記録がありません」等の空状態を表示すること。全問正解者のみ表示する文言や挙動は用いないこと。
5. 初回プレイデータは `leaderboardFirstPlay` を読み取ること。未移行クイズで当該フィールドが空のときのみ、旧 `leaderboard` フィールドを初回プレイ側のフォールバックとして表示してよい。
6. リプレイデータは `leaderboardReplay` のみを読み取ること。
7. E2Eおよびアクセシビリティ用に、クイズLB領域に `data-testid="quiz-leaderboard"` を付与し、初回側に `data-testid="highscore-leaderboard"`（または同等）、各行に `data-testid="leaderboard-entry"` を付与すること。
8. クイズ詳細画面は、リーダーボード配列の更新・並び替えロジックを実装してはならない（読み取りと表示のみ。更新は `quizeum-core`）。

### 要件 10: ジャンルマスタ駆動の探索UI（Phase 6）
**目的:** Quizeumのユーザーとして、運営が承認したジャンルマスタに基づいてホームやジャンル一覧を探索したい。それにより新設・マージ後のジャンルが画面に即時反映される。

#### 受け入れ基準
1. ホーム画面は、マウント時に `listActiveGenres`（または同等のコア API）で有効ジャンル一覧を取得し、表示名とアイコン画像 URL を用いたジャンルナビを描画すること。
2. ホーム画面のジャンルナビは、ソースとしてハードコード定数配列（例: `GENRES`）を**正本として用いてはならない**（「すべて」等の UI 専用オプションはローカル定義可）。
3. ジャンルアイコンをクリックしたとき、**常に** `/genres/[genreId]` へ遷移し、URL パラメータはマスタのジャンル ID（英小文字キー）を用いること（ホーム上でのインライン絞り込みにアイコンクリックを用いてはならない）。
4. ホームの複合検索パネルは、ジャンルサジェスト付きテキスト選択・難易度・問題数・プレイ状況の変更時に `searchQuizzes` を呼び出し（フィルタ変更トリガー、適切なデバウンス可）、すべての条件を AND で適用した結果をクイズグリッドに反映すること。プレイ状況は要件 1.3 と同規則とする。ジャンル・難易度・問題数・プレイ状況・キーワードがすべて未指定のときは従来のタブ別取得を維持してよい。
5. ジャンル別一覧画面（`/genres/[genreName]`）は、マスタから `displayName` と `iconImageUrl` を表示し、生の URL パラメータ文字列のみを見出しに用いないこと（マスタ未ヒット時は ID をフォールバック表示可）。
6. ジャンル別一覧画面は、「新着」「人気」「トレンド」のソートタブを提供し、選択に応じて `getQuizzesByGenre(genreId, limit, sort)` を呼び出すこと（C2 読み取りによりマージ済み旧ジャンルのクイズも含む）。
7. タグ別一覧画面（`/tags/[tagName]`）は、canonical 優先の `getQuizzesByTag` を用い、新着／人気／トレンドのソート切替に対応すること（既存 UI パターンと同型でよい）。
8. 弱点克服画面（`/quiz/review`）のジャンル選択は、ハードコード一覧の代わりに `listActiveGenres` 取得結果と「オールジャンル」を表示すること。
9. 有効ジャンルが 0 件のとき、ホームのジャンルナビは空状態または案内メッセージを表示し、クラッシュしないこと。
10. マスタ取得に失敗したとき、ホームおよびジャンル一覧はエラーまたは再試行可能な状態を表示し、サイレントにハードコード一覧へフォールバックしてはならないこと。

### 要件 11: 分類ブックマークと問題リストプレイ UI（Phase 8）
**目的:** 認証ユーザーとして、ブックマークしたクイズ・リスト・問題を種類ごとに管理し、問題リストを連続プレイしたい。それにより学習用のお気に入りと問題コレクションを活用できる。

#### 受け入れ基準
1. 認証済みユーザーがブックマーク一覧画面を開いたとき、プレイフロー UI は、「クイズ」「リスト」「問題」の3タブを表示し、選択中タブのみ該当カテゴリの一覧を表示すること。
2. 未認証ユーザーがブックマーク一覧画面にアクセスした場合、プレイフロー UI は、認証画面（`/login`）へ遷移すること。
3. 「クイズ」タブが選択されているとき、プレイフロー UI は、`getBookmarkFeed`（または `getBookmarkedQuizzes`）の結果をカード一覧で表示し、各カードからブックマーク解除トグルとクイズ詳細への遷移を提供すること。
4. 「リスト」タブが選択されているとき、プレイフロー UI は、ブックマークしたクイズリストをカード一覧で表示し、各カードからブックマーク解除トグルとリスト詳細（`/list/[id]`）への遷移を提供すること。
5. 「問題」タブが選択されているとき、プレイフロー UI は、ブックマークした問題を、問題文の抜粋・親クイズタイトル・ブックマーク日時の降順で一覧表示し、各カードからブックマーク解除トグルを提供すること。
6. ユーザーが問題タブの問題カードを選択したとき、プレイフロー UI は、当該問題の親クイズのプレイ画面（`/quiz/[parentQuizId]/play`）へ遷移し、当該問題から解答を開始できる導線を提供すること（単一問題プレイまたは当該問題インデックスへのスクロールでよい）。
7. 認証済みユーザーが通常または模擬試験プレイ中に問題を表示しているとき、プレイフロー UI は、当該問題のブックマーク登録／解除トグルを表示し、`toggleBookmark`（`targetType: 'question'`）を呼び出すこと。
8. 認証済みユーザーがクイズ結果画面で正誤一覧を表示しているとき、プレイフロー UI は、各問題行にブックマーク登録／解除トグルを表示し、親クイズが公開済みの問題のみ操作可能とすること。
9. 未認証ユーザーが問題ブックマーク操作を試みた場合、プレイフロー UI は、認証画面（`/login`）へ遷移すること。
10. ユーザーが `listType` が問題リストのリスト詳細画面（`/list/[id]`）を表示したとき、プレイフロー UI は、収録問題の順序付き一覧（問題文抜粋・親クイズタイトル）と「問題リストプレイ開始」ボタンを表示すること。
11. ユーザーが問題リストの「問題リストプレイ開始」を押したとき、プレイフロー UI は、リスト先頭の問題の親クイズプレイ画面へ遷移し、リスト内インデックスおよび `listId` をクエリまたはセッション状態で保持すること。
12. 問題リストプレイ中に1問題分の結果画面を完了したとき、プレイフロー UI は、リスト内に次の問題が残っている場合、次の問題のプレイ画面へ自動または明示ボタンで遷移し、最終問題完了後はリスト詳細または完了メッセージを表示すること。
13. ユーザーが `listType` がクイズリストのリスト詳細を表示したとき、プレイフロー UI は、従来どおり「リストプレイ開始」（クイズ単位連続プレイ）を提供し、問題リスト用 UI と混在させないこと。
14. プレイフロー UI は、`bookmarksCount` の更新や `attempts` の永続化ロジックを実装してはならない（`quizeum-core` のサービス呼び出しのみとする）。

### 要件 12: 探索検索のタグチップ化・サジェスト強化とクイズカード情報拡充（Phase 10）
**目的:** Quizeumのユーザーとして、ホームの検索バーでタグをチップ化しながらタグ名・ジャンル名のサジェストを受け、クイズ一覧カードから難易度・ジャンル・出題形式を一目で把握したい。それにより複数タグでの絞り込みが直感的になり、プレイ前の判断がしやすくなる。

#### 受け入れ基準

**検索バー — タグチップ**
1. ホーム画面の統合検索バーは、確定済みタグをチップ（バッジ）として検索入力の左側または内部に表示し、未確定の自由入力テキスト用フィールドを併設すること。
2. When ユーザーが検索入力中にスペースキーを押したとき、the Play Flow UI は、スペース直前の入力トークン（先頭の `#` は除去して正規化）をタグチップとして追加し、当該トークンを入力欄から除去すること。トークンが空または空白のみのときはチップを追加してはならない。
3. When ユーザーが検索入力中に Enter キーを押したとき、the Play Flow UI は、スペース確定と同一規則でタグチップを追加してよい（キーボード操作の代替確定として）。
4. When ユーザーが同一タグを重複してチップ化しようとしたとき、the Play Flow UI は、重複チップを追加してはならない。
5. When ユーザーがタグチップ上の削除（×）操作を行ったとき、the Play Flow UI は、当該チップを除去し、検索結果を再取得すること。
6. When ユーザーが消去（X）ボタンを押したとき、the Play Flow UI は、自由入力テキスト・すべてのタグチップ・検索バーに紐づくジャンル選択状態をクリアし、タブ別取得へ復帰してよいこと（フィルタパネル内の難易度・問題数・プレイ状況は従来どおり個別クリア可）。

**検索バー — タグ・ジャンルサジェスト**
7. While ユーザーが統合検索バーの自由入力欄に1文字以上入力している間、the Play Flow UI は、入力値に部分一致するタグ候補とジャンル候補をドロップダウン（またはリストボックス）で表示すること。ジャンル候補は表示名および ID の双方にマッチしてよい。
8. When ユーザーがサジェスト一覧からタグ候補を選択したとき、the Play Flow UI は、正規化済みタグ名のチップを追加し、自由入力欄をクリアすること。
9. When ユーザーがサジェスト一覧からジャンル候補を選択したとき、the Play Flow UI は、当該ジャンルを検索のジャンル条件として適用し（フィルタパネルのジャンル選択と同期してよい）、自由入力欄の既存テキストはクリアせずそのまま保持すること。
9a. When ジャンル条件が適用されているとき、the Play Flow UI は、検索バー内（または検索バーの直下）に「ジャンルで絞り込む：〈ジャンル表示名〉」の形式でジャンルバッジを表示し、バッジ内に ×ボタンを配置すること。
9b. When ユーザーがジャンルバッジ内の ×ボタンを押したとき、the Play Flow UI は、ジャンル条件のみをクリアし、自由入力テキスト・タグチップ・その他フィルタ条件は変更しないこと。
9c. The Play Flow UI は、ジャンルバッジの削除にフィルタパネル内のジャンル選択クリアと統合検索バー右端の全体消去（X）ボタンを用いてはならない（ジャンル条件のクリアはバッジ内の ×ボタンが唯一の正本とする）。
10. When タグマスタまたはジャンルマスタの取得に失敗したとき、the Play Flow UI は、サジェストを空またはエラー表示とし、サイレントにハードコード候補へフォールバックしてはならない。自由入力によるキーワード検索および既存フィルタは可能な範囲で継続してよい。

**検索バー — クイックサーチチップ連携**
11. When ユーザーが検索バー直下のクイックサーチチップ（例: `#ウミガメのスープ`）をクリックしたとき、the Play Flow UI は、プレーンテキストを入力欄へ流し込むのではなく、対応するタグチップを追加し、統合検索を即時実行すること。

**検索の実行 — チップとキーワードの合成**
12. When タグチップ・自由入力キーワード・フィルタパネル条件（ジャンル・難易度・問題数・プレイ状況）のいずれかが変更されたとき、the Play Flow UI は、デバウンス（300ms）後に `searchQuizzes` を呼び出し、すべての条件を AND で適用した結果をクイズグリッドに反映すること。
13. When 複数のタグチップが存在するとき、the Play Flow UI は、各タグ条件を AND（すべてのタグを含むクイズ）として検索すること。
14. When 自由入力キーワードが残っているとき、the Play Flow UI は、タイトル・説明・作者名等のテキスト部分一致検索にキーワードを用い、タグチップ条件と AND 合成すること。
15. When タグチップ・キーワード・ジャンル・難易度・問題数・プレイ状況がすべて未指定のとき、the Play Flow UI は、従来のタブ別取得（新着・人気・トレンド・タイムライン）を維持すること。

**クイズカード — 表示項目の拡充（ホーム・ジャンル一覧・タグ一覧共通）**
16. The Play Flow UI は、クイズ一覧で用いる共通クイズカードコンポーネントにおいて、難易度を **数値併記 of 星表記**（例: `★ 4`、難易度 1〜5 の整数値を星アイコンと併記）で表示すること。プログレスバー形式の難易度表示は用いてはならない。
17. The Play Flow UI は、各クイズカードにジャンル表示名を表示すること。ジャンルマスタから表示名を解決できるときはマスタの `displayName` を用い、未解決時はクイズに保存されたジャンル ID または名称をフォールバック表示してよい。
18. The Play Flow UI は、各クイズカードに出題形式の日本語ラベル（例: 選択式、記述式、ウミガメのスープ、複合形式 等）を表示すること。また、その表示名の直前には作問画面およびホーム出題形式カルーセルと同様の絵文字アイコンを併記すること（例: `🌀 複合形式`）。クイズに形式フィールドが無い旧データについては、問題構成から推定した形式ラベルとアイコンを表示してよい（推定・アイコン解決規則はコア／共有ユーティリティと一致させること）。
19. The Play Flow UI は、ホーム画面のクイズグリッド、ジャンル別一覧画面（`/genres/[genreName]`）、タグ別一覧画面（`/tags/[tagName]`）で、同一の共通クイズカードコンポーネントを用い、表示項目（難易度星表記・ジャンル・出題形式・評価・ブックマーク・プレイ導線）を揃えること。ジャンル／タグ一覧専用の重複インラインカード実装を正本として維持してはならない。
20. While クイズ一覧の検索またはソート結果の読み込み中である間、the Play Flow UI は、要件 1 のスケルトンスクリーンを表示し続けること。

**アクセシビリティ・テスト支援**
21. The Play Flow UI は、タグチップに削除可能であることを示すアクセシブル名を付与し、サジェスト一覧をキーボード（矢印・Enter）で操作可能にすること（既存ジャンルサジェストと同程度でよい）。
22. The Play Flow UI は、統合検索のタグチップ領域に `data-testid="search-tag-chips"`、各タグチップに `data-testid="search-tag-chip"`、サジェスト項目に `data-testid` プレフィックス（例: `search-suggest-tag-` / `search-suggest-genre-`）を付与すること。クイズカードの難易度表示に `data-testid="quiz-card-difficulty"`、ジャンルに `data-testid="quiz-card-genre"`、出題形式に `data-testid="quiz-card-format"` を付与すること。
23. The Play Flow UI は、ジャンルバッジに `data-testid="genre-filter-badge"`、バッジ内の ×ボタンに `data-testid="genre-filter-badge-clear"` を付与すること。

### 要件 13: 探索カルーセル・検索セクションおよびジャンルページ検索（Phase 11、Phase 21 でホーム UI 改定）
**目的:** Quizeumのユーザーとして、ホームでジャンルや出題形式から直感的に絞り込み、ジャンル別一覧ページでも当該ジャンル内を検索・フィルタしたい。それによりページ遷移なしで探索を完結し、深掘り時にも一貫した検索体験を得られる。

#### 受け入れ基準

**ホーム — 検索セクション内カルーセル（常時表示、Phase 21）**
1. ホーム画面は、統合検索バーおよびクイック検索チップを含む検索セクション（`ExploreSearchSection` 相当）内に、ジャンル絞り込み用の横スクロールカードカルーセルおよび出題形式絞り込み用の横スクロールカードカルーセルを**常時表示**すること。
2. The Play Flow UI shall [Phase 21 以降、ホーム画面に「ジャンルで絞り込む」「出題形式で絞り込む」のアコーディオン折りたたみ UI（`ExploreAccordion`）を**正本として表示してはならない**]。
3. The Play Flow UI shall [ジャンル・出題形式カルーセルを、「フィルター」ボタンで開閉する難易度・問題数・プレイ状況パネルの開閉対象に含めてはならない（カルーセルは常時表示を維持すること）]。
4. When [ジャンルカルーセルを表示するとき], the [Play Flow UI] shall [ジャンル名による絞り込み入力（`GenreSearchField` 相当）をカルーセル上部に表示してよいこと]。

**ホーム — ジャンルカードカルーセル**
5. The Play Flow UI は、`listActiveGenres` 由来の有効ジャンルを横スクロール可能なカードカルーセルとして表示すること。各カードはジャンル表示名およびアイコン（存在する場合）を含むこと。説明文がある場合は表示してよい。
6. When ユーザーがジャンルカードを選択したとき、the Play Flow UI は、**ページ遷移せず**当該ジャンルを探索フィルタ条件として適用し、デバウンス（300ms）後にクイズグリッドを更新すること。
7. When ジャンルがフィルタ条件として適用されているとき、the Play Flow UI は、対応するジャンルカードを選択中状態（ハイライト等）で表示すること。
8. When ユーザーが選択中のジャンルカードを再度選択したとき、the Play Flow UI は、ジャンルフィルタ条件を解除し、他のフィルタ条件が未指定ならタブ別取得へ復帰してよいこと。
9. When ユーザーが統合検索バーのジャンルサジェストからジャンルを選択したとき、the Play Flow UI は、ジャンルカルーセルの選択状態と同期すること。

**ホーム — 出題形式カードカルーセル**
10. The Play Flow UI は、プラットフォームで利用可能な出題形式（複合形式、選択式、記述式、早押し、並び替え、連想、ウミガメのスープ、〇×式 等）を横スクロール可能なカードカルーセルとして表示すること。各カードは作問画面と同様の絵文字アイコンおよび日本語ラベルを併記して表示すること。
11. When ユーザーが出題形式カードを選択したとき、the Play Flow UI は、**ページ遷移せず**当該出題形式を探索フィルタ条件として適用し、デバウンス（300ms）後にクイズグリッドを更新すること。
12. When 出題形式がフィルタ条件として適用されているとき、the Play Flow UI は、対応する出題形式カードを選択中状態で表示すること。
13. When ユーザーが選択中の出題形式カードを再度選択したとき、the Play Flow UI は、出題形式フィルタ条件を解除すること。

**ホーム — 既存ジャンルピルナビの置換**
14. The Play Flow UI shall [Phase 11 以降、ホーム画面に要件 1.2 および要件 10.1–10.3 で定義されたジャンル「ピル」横スクロールナビ（`GenreNav`）を**正本として表示してはならない**。ジャンル探索の正本 UI は本要件の検索セクション内カルーセルとする]。
15. When ユーザーがホームのジャンルカルーセルからジャンルを選択したとき、the Play Flow UI shall [`/genres/[genreId]` へ遷移してはならない（ホーム内フィルタ型を維持すること）]。

**ホーム — フィルタ状態の共有と検索実行**
16. The Play Flow UI shall [統合検索バー（タグチップ・キーワード・ジャンルサジェスト）、展開可能なフィルタパネル（難易度・問題数・プレイ状況等）、ジャンルカルーセル、出題形式カルーセルが、同一の探索フィルタ状態を共有すること]。
17. When 探索フィルタ条件（タグチップ・キーワード・ジャンル・出題形式・難易度・問題数・プレイ状況のいずれか）が変更されたとき、the Play Flow UI は、デバウンス（300ms）後に先頭ページから一覧を再取得し、すべての条件を AND で適用した結果をクイズグリッドに反映すること（要件 21 と整合）。
18. When すべての探索フィルタ条件が未指定のとき、the Play Flow UI は、従来のタブ別取得（新着・人気・トレンド・タイムライン）を維持すること。
19. When ユーザーが統合検索バーの一括クリア操作を行ったとき、the Play Flow UI は、ジャンルカルーセルおよび出題形式カルーセルの選択状態も解除すること。

**ジャンル別一覧ページ — 検索・フィルタ**
20. ジャンル別一覧画面（`/genres/[genreName]`）は、見出しおよびソートタブの上または下に、ホーム画面と同型の統合検索バーおよび展開可能なフィルタパネルを表示すること。
21. The Play Flow UI shall [ジャンル別一覧ページでは、URL パラメータで指定されたジャンルを探索対象として固定し、フィルタパネル内のジャンルセレクトは非表示または読み取り専用とすること。ユーザーが当該ページ上で他ジャンルへ切り替えてはならない]。
22. When ジャンル別一覧ページで検索またはフィルタ条件が変更されたとき、the Play Flow UI は、固定ジャンルを含む scoped 条件で `searchQuizzes`（または同等の複合検索 API）を呼び出し、当該ジャンル内のクイズのみをグリッドに反映すること。
23. When ジャンル別一覧ページでキーワード・タグ・出題形式・難易度・問題数等がすべて未指定のとき、the Play Flow UI は、従来どおりソートタブ（新着・人気・トレンド）に応じた `getQuizzesByGenre` 取得を維持してよいこと。
24. While ジャンル別一覧ページで scoped 検索結果の読み込み中である間、the Play Flow UI は、スケルトンスクリーンを表示すること。

**境界・隣接**
25. The Play Flow UI shall [出題形式専用一覧ルート（`/formats/[format]`）の新設、URL クエリによるフィルタ共有可能化、タグ別一覧ページへの検索バー追加を本要件の範囲に含めない]。
26. The Play Flow UI shall [出題形式フィルタのサーバー側照合ロジックを実装してはならない（`quizeum-core` の `searchQuizzes` 拡張に依存する）]。

**アクセシビリティ・テスト支援**
27. The Play Flow UI shall [ジャンルカルーセルに `data-testid="genre-carousel"`、出題形式カルーセルに `data-testid="format-carousel"`、各カードに `data-testid` プレフィックス（例: `genre-carousel-card-` / `format-carousel-card-`）を付与すること]。
28. When ジャンル別一覧ページに scoped 検索 UI を表示するとき、the Play Flow UI shall [`data-testid="genre-explore-search"` を検索領域に付与すること]。

### 要件 14: 検索フィールドのフォーカス時スマートサジェスト（Phase 10 追加）
**目的:** Quizeumのユーザーとして、ジャンル検索および統合検索のフィールドをフォーカスしたとき、自分の直近検索履歴や週間の人気トレンドに基づく候補を即座に確認したい。それにより空クエリ状態でも効率よく検索を始められる。

#### 受け入れ基準

**ジャンル検索フィールド — フォーカス時スマートサジェスト**
1. When [ユーザーがジャンル検索フィールドをフォーカスし、入力欄が空のとき], the [Play Flow UI] shall [直近検索ジャンルのセクション（最大3件）と週間プレイ数の多いジャンルのセクション（Top5）を順にドロップダウン内に表示する。入力があれば従来の部分一致サジェストに切り替わる]。
2. When [ジャンル検索フィールドに空クエリ時のドロップダウンを表示するとき], the [Play Flow UI] shall [「最近検索したジャンル」と「今週の人気ジャンル」の2セクションラベルを表示し、各アイテムにジャンル表示名を表示する]。
3. When [ユーザーの直近検索ジャンル履歴が存在するとき], the [Play Flow UI] shall [最大3件を直近検索ジャンルセクションとして表示する]。
4. When [ユーザーの直近検索ジャンル履歴が1件も存在しないとき], the [Play Flow UI] shall [直近検索ジャンルセクションを表示しない]。
5. When [週間人気ジャンルデータの取得中であるとき], the [Play Flow UI] shall [週間人気ジャンルセクションにローディング状態を表示する]。
6. If [週間人気ジャンルデータの取得に失敗した場合], the [Play Flow UI] shall [週間人気ジャンルセクションを非表示またはエラー表示とし、代替として全ジャンル一覧をサイレントに表示してはならない]。
7. When [ユーザーがスマートサジェストドロップダウンからジャンルを選択したとき], the [Play Flow UI] shall [要件 12.9 と同一の挙動（ジャンル条件の適用・入力テキスト保持・ジャンルバッジ表示）を実行し、選択されたジャンルをブラウザローカルの直近検索ジャンル履歴に追加すること。この操作はサーバーに送信しない]。
8. When [直近検索ジャンル履歴に3件未満のエントリがあるとき], the [Play Flow UI] shall [存在する件数のみ表示し、不足分をプレースホルダーで埋めない]。

**統合検索フィールド — フォーカス時スマートサジェスト**

### 要件 16: 結果画面の解説アコーディオン化と難易度投票UI改善（Phase 14）
**目的:** クイズプレイヤーとして、結果画面で正誤一覧を素早く把握しつつ、必要な問題だけ回答と解説を展開して確認したい。また体感難易度は直感的な★操作で投票したい。それにより長いクイズでも結果画面が読みやすくなり、フィードバック入力がしやすくなる。

#### 受け入れ基準

**問題ごとの回答・解説アコーディオン**
1. When [ユーザーが本番クイズ結果画面（`/quiz/[id]/result`）で問題ごとの正誤一覧を表示したとき], the [Play Flow UI] shall [各問題について、第N問・正誤ラベル・問題文、および問題行の操作ボタン（ブックマーク・指摘等）を常時表示すること]。
2. When [各問題行を表示したとき], the [Play Flow UI] shall [「あなたの回答」「正解」の回答サマリーおよびマークダウン解説を、アコーディオン（折りたたみ）領域内に配置し、初期状態では折りたたんだ（閉じた）状態で表示すること]。
3. When [ユーザーが問題行のアコーディオン見出し（または同等のトグル操作）を操作したとき], the [Play Flow UI] shall [当該問題の回答サマリーおよび解説の展開／折りたたみを切り替え、展開状態を視覚的に示すこと]。

5. Where [出題形式が連想クイズ（association）またはウミガメのスープ（lateral-thinking）であり、結果詳細としてヒント履歴または質問回数を表示するとき], the [Play Flow UI] shall [当該情報も回答・解説と同一のアコーディオン領域内に含め、初期状態では折りたたんだ状態とすること]。
6. When [ユーザーがテストプレイ結果画面（`/quiz/test-play/result`）で問題ごとの正誤一覧を表示したとき], the [Play Flow UI] shall [本番結果画面と同一規則で回答・解説のアコーディオン表示（初期は閉じた状態）を適用すること]。

8. When [ユーザーが体感難易度の★アイコン（第N個、Nは1〜5）をクリックしたとき], the [Play Flow UI] shall [Nを体感難易度の投票値として登録し、選択中の★表示を更新すること]。

10. When [ユーザーが既に体感難易度を投票済みであるとき], the [Play Flow UI] shall [投票済みの値に対応する★数を塗りつぶし表示し、残りを空星（☆）で表示すること]。
11. While [ネットワーク接続がオフラインである間], the [Play Flow UI] shall [体感難易度の★アイコンを非活性とし、クリックによる投票を受け付けないこと（要件5.5と整合）]。

### 要件 17: 通常モード即時フィードバック・スキップ・楽観的結果遷移（Phase 15）
**目的:** クイズプレイヤーとして、通常モードで回答後に正誤と解説を確認してから次に進みたい。わからない問題はスキップし、最後は待たずに結果画面を見たい。それにより学習効果を高め、完了時のストレスを減らせる。

#### 受け入れ基準

**対象モードとスコープ**
1. The [Play Flow UI] shall [本要件のフィードバックフロー・スキップ・楽観的結果遷移を、本番プレイ画面（`/quiz/[id]/play`）における通常モード（`mode=normal`）にのみ適用すること]。
2. The [Play Flow UI] shall [模擬試験（`mode=exam`）、フラッシュカード（`mode=flashcard`）、ウミガメのスープ（`mode=lateral`）、問題リストプレイ（`mode=question-list`）については、本要件の変更前のプレイ進行挙動を維持すること]。
3. The [Play Flow UI] shall [テストプレイ画面（`/quiz/test-play/play`）を本要件の対象外とすること]。

**回答後の即時正誤フィードバック**
4. When [通常モードでユーザーが問題に回答を送信したとき], the [Play Flow UI] shall [その場で正誤判定結果（正解／不正解）を表示し、自動的に次の問題へ進んではならない]。
5. When [正誤表示を行ったとき], the [Play Flow UI] shall [問題に解説が設定されている場合は、同一フィードバック領域内に解説を表示すること]。

7. The [Play Flow UI] shall [通常モードの全対応問題形式（選択式、〇×式、記述式、早押し、並び替え、連想）について、同一のフィードバックフローを適用すること]。
8. When [通常モードで制限時間が0秒に達したとき], the [Play Flow UI] shall [空回答と同等に不正解として記録し、即時正誤フィードバックを表示すること（要件 3.1 と整合）]。

**スキップボタン**
9. While [通常モードで現在の問題が未解答（フィードバック待ち状態を含む）である間], the [Play Flow UI] shall [「わからない（スキップ）」ボタン（または同等ラベル）を表示すること]。
10. When [ユーザーがスキップボタンを押したとき], the [Play Flow UI] shall [空回答（`''`）を提出した場合と同一規則で不正解として記録し、`failedQuestionIds` に当該問題を含めること]。
11. When [ユーザーがスキップボタンを押したとき], the [Play Flow UI] shall [即時正誤フィードバック（不正解）を表示し、自動的に次の問題へ進んではならない]。

**次へ／結果を見る**
12. When [通常モードで正誤フィードバックが表示されているとき], the [Play Flow UI] shall [「次へ」ボタンを表示すること（最終問題を除く）]。
13. When [通常モードで最終問題の正誤フィードバックが表示されているとき], the [Play Flow UI] shall [「次へ」ボタンの代わりに「結果を見る」ボタンを表示すること]。
14. When [ユーザーが「次へ」ボタンを押したとき], the [Play Flow UI] shall [次の未解答問題へ進み、フィードバック表示状態をリセットすること]。
15. When [ユーザーが最終問題で「結果を見る」ボタンを押したとき], the [Play Flow UI] shall [プレイセッションを完了扱いとし、結果画面へ遷移すること]。

**楽観的結果遷移（送信中画面の廃止）**
16. When [通常モードでユーザーが「結果を見る」を押したとき], the [Play Flow UI] shall [「解答データを送信中...」等の中間待機画面を表示せず、直ちに結果画面（`/quiz/[id]/result`）へ遷移すること]。
17. When [通常モード完了後に結果画面へ遷移したとき], the [Play Flow UI] shall [結果画面の静的フレームを即時表示し、結果コンテンツ領域には要件 15 の `ResultSkeleton` を表示すること（要件 5.1a と整合）]。
18. When [通常モード完了後にオンラインであるとき], the [Play Flow UI] shall [結果画面への遷移を永続化完了より優先し、`saveAttempt` をバックグラウンドで実行すること。永続化の成否は結果画面側でユーザーに適切に反映してよい]。
19. When [通常モード完了後にオフラインであるとき], the [Play Flow UI] shall [既存のオフライン結果遷移（`localId` クエリおよび pending sync）を維持しつつ、中間待機画面を表示しないこと]。

**クイズ詳細の即時正誤トグル**
20. Where [プレイモードが通常モードであるとき], the [Play Flow UI] shall [クイズ詳細画面の「解答後にその場で正誤・解説を表示」トグル設定にかかわらず、本要件の即時フィードバックフローを常に適用すること]。

**境界・隣接**
21. The [Play Flow UI] shall [`saveAttempt` の API 契約・スキーマ変更を本要件の範囲に含めない（`quizeum-core` の既存 API を呼び出すのみ）]。
22. The [Play Flow UI] shall [正誤判定ロジックの正本を `quizeum-core` / 共有ユーティリティに移さず、既存のクライアント側判定規則を維持すること]。

**アクセシビリティ・テスト支援**
23. The [Play Flow UI] shall [通常モードの正誤フィードバック領域に `data-testid="play-answer-feedback"`、スキップボタンに `data-testid="play-skip-question"`、「次へ」ボタンに `data-testid="play-next-question"`、「結果を見る」ボタンに `data-testid="play-view-results"` を付与すること]。

### 要件 18: 早押し形式の経過時間・制限時間・フィードバック・レイアウト（Phase 16）
**目的:** クイズプレイヤーとして、早押し形式の問題を公平なタイム計測と見やすいレイアウトでプレイしたい。不正解時に正解を知らされず、問読み前や制限時間終了後に時間が加算されないようにしたい。それにより早押し競技として信頼できる体験を得られる。

#### 受け入れ基準

**対象とスコープ**
1. The [Play Flow UI] shall [本要件を、本番プレイ画面（`/quiz/[id]/play`）における通常モード（`mode=normal`）かつ現在の問題が早押し形式（`quick-press`）である場合に適用すること]。
2. The [Play Flow UI] shall [模擬試験・フラッシュカード・ウミガメのスープ・問題リストプレイ・テストプレイ画面を本要件の対象外とすること]。
3. The [Play Flow UI] shall [早押しタイム（「押して回答する！」押下から回答送信までの秒数）の計測・正解時のみの記録・結果画面への統計表示について、既存仕様を維持すること]。

**セッション経過時間 — 区間累計**
4. The [Play Flow UI] shall [プレイ画面上部に表示する経過時間（⏱️）を、プレイ開始からの壁時計ではなく、各問題で計測した区間の累計秒数として表示すること]。
5. When [通常モードのプレイが完了し `elapsedSeconds` を記録するとき], the [Play Flow UI] shall [画面上部表示と同一規則の区間累計秒数を `elapsedSeconds` として保存すること（`quizeum-core` の既存 `saveAttempt` フィールドを利用）]。
6. Where [セッションに早押し形式以外の問題が含まれる場合], the [Play Flow UI] shall [早押し形式以外の問題については従来の経過時間計測（要件 3）を維持しつつ、セッション合計表示は各問題区間の累計とすること]。

**早押し問題 — 区間計測の開始・継続・停止**
7. While [早押し形式の問題が表示されており、ユーザーが「問読みを開始する」をまだ押していない間], the [Play Flow UI] shall [当該問題の経過時間区間を加算してはならない]。
8. When [ユーザーが早押し形式の問題で「問読みを開始する」を押したとき], the [Play Flow UI] shall [当該問題の経過時間区間の加算を開始すること]。
9. While [早押し形式の問題で問読みが進行中（問読み開始から問読み修了まで）である間], the [Play Flow UI] shall [当該問題の経過時間区間を加算し続けること]。
10. When [早押し形式の問題の問読みが修了したとき], the [Play Flow UI] shall [当該問題に制限時間（`limitTime`）が設定されている場合、問読み修了後に制限時間のカウントダウンを開始すること（問読み開始前および問読み中はカウントダウンを開始してはならない）]。
11. While [早押し形式の問題で問読み修了後の制限時間カウントダウンが進行中である間], the [Play Flow UI] shall [当該問題の経過時間区間を加算し続けること]。
12. When [早押し形式の問題で制限時間のカウントダウンが 0 秒に達したとき], the [Play Flow UI] shall [当該問題の経過時間区間の加算を停止し、空回答と同等に不正解として記録し、即時正誤フィードバックを表示すること（要件 3.1・要件 17.8 と整合）]。
13. When [早押し形式の問題でユーザーが不正解（誤った回答の送信、スキップ、制限時間切れを含む）が確定したとき], the [Play Flow UI] shall [当該問題の経過時間区間の加算を停止し、確定時点までの区間をセッション累計に確定すること]。
14. While [早押し形式の問題で不正解が確定し、ユーザーが「次へ」または「結果を見る」を押すまでのフィードバック表示中である間], the [Play Flow UI] shall [経過時間区間を加算してはならない]。
15. When [早押し形式の問題でユーザーが正解を送信したとき], the [Play Flow UI] shall [当該問題の経過時間区間の加算を停止し、確定時点までの区間をセッション累計に確定すること]。
16. When [ユーザーが「次へ」により次の問題へ進んだとき], the [Play Flow UI] shall [次の問題の種別に応じた経過時間規則（早押し形式なら本要件、それ以外なら要件 3）で区間計測を再開または待機すること]。

**不正解フィードバック — 正解非表示**
17. When [早押し形式の問題で不正解の即時フィードバックを表示するとき], the [Play Flow UI] shall [フィードバック領域内に正解内容（または正解の要約）を表示してはならない]。
18. When [早押し形式の問題で不正解の即時フィードバックを表示するとき], the [Play Flow UI] shall [問題に解説が設定されている場合は、解説を同一フィードバック領域内に表示してよいこと（要件 17.5 と整合）]。

**問題カードレイアウト**
19. When [通常モードで早押し形式の問題を表示するとき], the [Play Flow UI] shall [「問読みを開始する」操作前から、問題カード（問題文表示領域を含むメインプレイエリア）を十分な横幅で表示すること。問読み開始後にのみカード幅が拡大する挙動を用いてはならない]。
20. The [Play Flow UI] shall [早押し形式の問題文表示における既存のストリーム演出（プレフィックス「問題：」、本文の逐次表示、強調表示）を維持すること]。

**境界・隣接**
21. The [Play Flow UI] shall [`saveAttempt` の API 契約・`elapsedSeconds` フィールドのスキーマ変更を本要件の範囲に含めない（区間累計値を既存フィールドへ格納するのみ）]。
22. The [Play Flow UI] shall [弱点克服（復習）プレイ画面の早押し表示・計測仕様を本要件の範囲に含めない]。
23. The [Play Flow UI] shall [リーダーボードの合計解答時間の集計・順位規則を実装してはならない（`quizeum-core` が `elapsedSeconds` を参照する）]。

**アクセシビリティ・テスト支援**
24. The [Play Flow UI] shall [早押し形式の「問読みを開始する」ボタンに `data-analytics="quiz-quickpress-reading-start"` を維持し、プレイ画面上部の経過時間表示領域をテストで識別可能にすること（例: 進行状況テキスト内の経過時間ラベル、または同等の `data-testid`）]。

### 要件 19: 模擬試験・フラッシュカードのランキング非対象警告（Phase 19）
**目的:** クイズプレイヤーとして、模擬試験・フラッシュカードがランキング対象外であることと、先にこれらでプレイすると初回プレイランキングに載れなくなることを、プレイ開始前に知りたい。それにより意図しない初回プレイ権利の喪失を防げる。

#### 受け入れ基準

**警告の表示条件**
1. When [ユーザーがクイズ詳細画面のプレイモード選択パネルを表示し、模擬試験・フラッシュカードの選択 UI が表示されているとき], the [Play Flow UI] shall [ランキング非対象に関する警告文を表示すること]。
2. Where [出題形式により模擬試験・フラッシュカードの選択 UI が非表示（通常モード固定）である場合], the [Play Flow UI] shall [本要件の警告を表示しない]。
3. Where [水平思考（ウミガメのスープ）専用の単一モード表示である場合], the [Play Flow UI] shall [本要件の警告を表示しない]。

**警告の内容**
4. The [Play Flow UI] shall [警告文に、模擬試験モードおよびフラッシュカードモードの完了記録はクイズ単位リーダーボード（初回プレイ・リプレイ）に掲載されない旨を含めること]。
5. The [Play Flow UI] shall [警告文に、模擬試験またはフラッシュカードモードで先にプレイした場合、以降に通常モードでプレイしても初回プレイランキングには掲載されない（リプレイランキングのみ対象となりうる）旨を含めること]。
6. The [Play Flow UI] shall [警告文を、模擬試験・フラッシュカード各モードの説明文付近、またはプレイパネル全体で共通の注意として、ユーザーがプレイ開始前に目に入る位置に配置すること]。

**境界・隣接**
7. The [Play Flow UI] shall [リーダーボードへの登録判定・初回／リプレイ振り分けロジックを実装してはならない（`quizeum-core` の要件 9 に依存する）]。
8. The [Play Flow UI] shall [プレイ画面・結果画面でのランキング非対象に関する追加警告を本要件の範囲に含めない（詳細画面のモード選択時の事前告知に限定する）]。

**アクセシビリティ・テスト支援**
9. The [Play Flow UI] shall [ランキング非対象警告領域に `data-testid="play-mode-leaderboard-warning"` を付与すること]。

### 要件 20: 〇×問題の1タップ回答 UI（Phase 20）
**目的:** クイズプレイヤーとして、〇×問題に対し 〇 または × を押すだけで即座に回答したい。それにより素早く直感的にプレイできる。

#### 受け入れ基準

**専用回答 UI**
1. When [現在の問題タイプが `true-false` であるとき], the [Play Flow UI] shall [選択式用の `ChoiceAnswerPanel`（ラジオ選択＋「解答を確定する」）を表示せず、専用の 〇／× 回答 UI を表示すること]。
2. The [Play Flow UI] shall [専用回答 UI に、視認性の高い「〇」ボタンと「✕」（または「×」）ボタンの2つを並べて表示すること]。
3. When [ユーザーが「〇」または「✕」ボタンを1回押したとき], the [Play Flow UI] shall [追加の確定操作なしで、直ちに当該選択肢を解答として送信すること]。
4. While [解答送信処理中または通常モードで正誤フィードバック表示中である間], the [Play Flow UI] shall [〇／× ボタンを無効化し、二重送信を防止すること]。

**対象画面とモード**
5. The [Play Flow UI] shall [本番プレイ画面（`/quiz/[id]/play`）の `true-false` 問題に本要件を適用すること]。
6. The [Play Flow UI] shall [テストプレイ画面（`/quiz/test-play/play`）の `true-false` 問題に、本番と同一の 〇／× 専用回答 UI を適用すること]。
7. The [Play Flow UI] shall [弱点克服プレイ画面（`/quiz/review`）の `true-false` 問題に、本番と同一の 〇／× 専用回答 UI を適用すること]。
8. Where [プレイモードが通常モード（`mode=normal`）である場合], the [Play Flow UI] shall [1タップ送信後、要件 17 に定義する即時正誤フィードバックフロー（正誤表示 →「次へ」／「結果を見る」）を適用すること]。
9. Where [プレイモードが模擬試験（`mode=exam`）である場合], the [Play Flow UI] shall [1タップ送信後、従来の模擬試験進行（確定後の自動遷移なし、見直し可能）を維持すること]。

**形式ラベルと探索**
10. The [Play Flow UI] shall [クイズカードおよび出題形式表示において、有効な出題形式 `true-false` の日本語ラベルを「〇×式」として表示すること]。
11. When [ホームの出題形式カルーセル（要件 13）を表示するとき], the [Play Flow UI] shall [「〇×式」を選択可能な形式カードとして含めること]。
12. When [ユーザーが出題形式カルーセルで「〇×式」を選択したとき], the [Play Flow UI] shall [ホーム内クイズグリッドを `true-false` 形式のクイズのみに絞り込むこと（`quizeum-core` の形式フィルタと整合）]。

**境界・隣接**
13. The [Play Flow UI] shall [`true-false` 問題の公開検証・選択肢正規化・`Quiz.format` 解決を実装してはならない（`quizeum-core` が担当）]。
14. The [Play Flow UI] shall [作問エディタの正解トグル UI を本要件の範囲に含めない（`quizeum-creator-dash-ui` が担当）]。
15. The [Play Flow UI] shall [正誤判定ロジックの正本を変更せず、既存の選択肢 ID ベース判定を維持すること]。

**アクセシビリティ・テスト支援**
16. The [Play Flow UI] shall [〇ボタンに `data-testid="true-false-answer-true"`、✕ボタンに `data-testid="true-false-answer-false"`、専用回答 UI のコンテナに `data-testid="true-false-answer-panel"` を付与すること]。

### 要件 21: ホームフィード無限スクロール・sticky 検索バー・フィルタ UI 再編（Phase 21）
**目的:** Quizeumのユーザーとして、ホームでクイズを探すとき初回表示を軽くし、スクロールしながら続きを読み込みたい。また、スクロール中も検索バーにすぐアクセスしたい。それにより探索が途切れず、絞り込み操作もしやすくなる。

#### 受け入れ基準

**初回取得と追加読み込み**
1. When [ユーザーがホーム画面を表示したとき], the [Play Flow UI] shall [クイズ一覧の初回ページのみを取得して表示すること（全件一括取得を正本として維持してはならない）]。
2. When [ユーザーがクイズグリッドを下方向にスクロールし、一覧末端付近に到達したとき], the [Play Flow UI] shall [自動的に次ページを取得し、既存のクイズカードの末尾に追加表示すること]。
3. While [次ページの取得中である間], the [Play Flow UI] shall [一覧末尾に追加読み込み用のローディング表示（スケルトンまたはスピナー）を表示すること]。
4. When [続きのクイズが存在しないとき], the [Play Flow UI] shall [追加の自動読み込みを行わないこと]。
5. When [続きのクイズが存在しないとき], the [Play Flow UI] shall [「これ以上ありません」等の末尾メッセージを表示してもよいが、誤解を招く無限ローディングを表示してはならない]。

**タブ・検索・フィルタとの整合**
6. When [ユーザーがタブ（新着・人気・トレンド・フォロー TL）を切り替えたとき], the [Play Flow UI] shall [クイズ一覧をリセットし、当該タブの先頭ページから再取得すること]。
7. When [探索フィルタ条件（タグチップ・キーワード・ジャンル・出題形式・難易度・問題数・プレイ状況のいずれか）が変更されたとき], the [Play Flow UI] shall [デバウンス（300ms）後にクイズ一覧をリセットし、先頭ページから再取得すること]。
8. When [タブ別取得または複合検索の追加ページを要求するとき], the [Play Flow UI] shall [`quizeum-core` の段階的取得 API を用い、同一条件での続きページを要求すること]。
9. While [プレイ状況フィルタ（未プレイ／プレイ済み）が有効で、表示件数が少ないが続きが存在するとき], the [Play Flow UI] shall [ユーザーが実質的に探索を続けられるよう、追加読み込みを継続してよいこと]。

**sticky 検索バー**
10. When [ユーザーがホーム画面で縦方向にスクロールしたとき], the [Play Flow UI] shall [検索バー行（統合検索フィールドと「フィルター」ボタンを含む行）を画面上部に固定表示すること]。
11. The [Play Flow UI] shall [sticky 固定の対象を検索バー行に限定し、クイック検索チップ行・ジャンル／出題形式カルーセル・展開中のフィルタパネル全体を sticky 対象に含めてはならない]。
12. While [検索バー行が sticky 表示されている間], the [Play Flow UI] shall [共通ナビゲーション（サイドバー・ボトムナビ）と視覚的に競合しないよう、固定表示が崩れないこと]。

**フィルタ UI 再編（要件 13 との整合）**
13. The [Play Flow UI] shall [ホーム画面から独立した `ExploreAccordionsPanel`（アコーディオン付きジャンル／出題形式パネル）を表示してはならない]。
14. The [Play Flow UI] shall [ジャンル・出題形式の横スクロールカルーセルを検索セクション内に常時表示すること（要件 13 と整合）]。

**境界・隣接**

# 要件定義書: quizeum-play-flow-ui

## はじめに
本ドキュメントは、クイズ投稿SNS「quizeum」におけるホーム画面、クイズ探索、クイズ詳細、プレイ画面（通常・ウミガメのスープ含む）、結果画面、弱点克服、およびリーダーボードを含む、ユーザーのクイズプレイに関するフロントエンドUI要件を定義します。

**Phase 5（2026-06）**: クイズ詳細のリーダーボード表示を、初回プレイ／リプレイの二系統に改定し、正解数優先・同点時タイムの表示規則に合わせる（データ更新は `quizeum-core` が担当）。

**Phase 6（2026-06）**: ジャンルマスタ駆動の探索 UI。アイコンクリックは `/genres/[id]` 遷移に統一。複合検索はサジェスト付きジャンル入力・フィルタ変更時の `searchQuizzes`（デバウンス）・認証ユーザーのプレイ状況絞り込み（要件 1.3 完遂）を含む。ハードコード `GENRES` 廃止、ジャンル／タグ一覧にソートタブ追加。

**Phase 8（2026-06）**: ブックマーク画面をクイズ・リスト・設問の3タブに拡張し、プレイ／結果画面からの設問ブックマーク、および設問リスト（`listType: 'question'`）の連続プレイ導線を追加する（データ取得・永続化は `quizeum-core` が担当）。
**Phase 9（2026-06）**: トップページレイアウトのスタイリッシュ化、単一の統合検索バーによるクイズ名・作者・タグ・ジャンルのハイブリッド検索、クイックサーチチップ、検索中スケルトンプレースホルダーの導入。

**Phase 10（2026-06）**: ホーム統合検索のタグチップ化（スペース確定）とタグ・ジャンル名サジェスト、クイズ一覧カードの難易度を数値併記の星表記（例: ★ 7）へ変更しジャンル・出題形式を表示、ジャンル／タグ一覧ページのカード表示をホームと統一する（タグマスタ読み取りは `quizeum-core` が担当）。

## 境界コンテキスト
- **対象範囲（In scope）**:
  - ホーム画面における新着・人気・トレンド・フォロータイムラインのタブ表示および複合検索。
  - クイズ詳細画面における良問評価バッジ、**初回プレイ／リプレイの二系統リーダーボード**、3つのプレイモード選択UI、および作成者本人用の編集ボタン表示と編集画面遷移機能。
  - クイズ編集画面（`/quiz/[id]/edit`）における他ユーザー（非作成者）による直接アクセス時の認可保護ガード処理。
  - 通常プレイ画面における設問表示、制限時間タイマー、ヒント表示、および `localStorage` を用いたセッション保護。
  - 水平思考（ウミガメのスープ）における2カラムレイアウト、AIチャット（入力制限、ターン制限、回答生成中の「・・・AIが質問を分析中です」グレー表示、キャッシュマーク）、および真相判定。
  - クイズ結果画面における正誤解説リスト、👍/👎良問評価、難易度投票、指摘フォーム、お礼リアクション、SNS共有。
  - 過去の間違い設問を復習できる弱点克服プレイ画面（ジャンル選択機能含む）。
  - 各種探索一覧画面（ブックマーク、タグ別、ジャンル別）および**プラットフォーム総合**リーダーボード画面（`/leaderboard`）。
  - **Phase 8**: ブックマーク一覧の3分類タブ（クイズ／リスト／設問）、プレイ・結果画面での設問ブックマーク操作、リスト詳細における設問リスト連続プレイ開始と次設問への遷移。
  - **Phase 9**: 統合検索バーおよびクイックサーチバッジチップ、ローディング時のカードスケルトンスクリーン、検索フォーカス時・ホバー時のネオン調インタラクション。
  - **Phase 10**: ホーム検索バー内のタグチップ入力・タグ／ジャンルサジェスト、クイズカードの難易度星表記（数値併記）・ジャンル・出題形式表示、ジャンル／タグ一覧での共通クイズカード利用。
- **対象外（Out of scope）**:
  - Gemini APIを利用した判定サーバーサイドロジック、リーダーボード永続化、`attempts` への書き込み（`quizeum-core` が担当）。
  - クイズ作成者向けのクイズ・クイズリスト作成・編集画面（`quizeum-creator-dash-ui` が担当）。
  - マイページのプレイ履歴一覧（`quizeum-auth-profile-ui` が担当）。
  - プラットフォーム総合リーダーボード（`/leaderboard`）の集計ロジック変更。
  - **Phase 8**: リストの作成・編集・`listType` 切替・設問のリストへの追加 UI（`quizeum-creator-dash-ui`）。作問エディタの過去クイズ検索・参照リンク UI（`quizeum-creator-dash-ui`）。設問を「マイリストに追加」するピッカー UI（リスト編集画面が正本。プレイ画面からの直接追加は初版対象外とする）。
  - **Phase 10**: ジャンル／タグ一覧ページへの検索バー新設、タグ新設申請・マージ UI の変更、クイズ詳細・プレイ画面の難易度表示変更、サーバー側ファジーサジェスト専用 API の新設。
- **隣接システムへの期待**:
  - クイズドキュメントの `leaderboardFirstPlay` / `leaderboardReplay` を読み取り表示する。`leaderboardFirstPlay` が空で旧 `leaderboard` のみ存在する場合は、初回プレイ側のフォールバック表示として旧フィールドを用いる（`quizeum-core` と同一ルール）。
  - **Phase 6**: ジャンルマスタ読み取り・一覧クエリ・canonical 解決は `quizeum-core`（`listActiveGenres`, `getQuizzesByGenre`, `getQuizzesByTag`, `searchQuizzes`）に依存する。エディタの動的ジャンルセレクトは `quizeum-creator-dash-ui` が担当。
  - **Phase 8**: 分類ブックマーク一覧は `getBookmarkFeed`（または同等の3分類取得 API）、設問ブックマークトグルは `toggleBookmark`（`targetType: 'question'`）、設問リストのメンバー解決と順序は `getQuestionsInList` に依存する。`attempts` への `mode: 'question-list'` 記録は `quizeum-core` の `saveAttempt` が担当する。
  - **Phase 10**: タグサジェスト用の有効タグ一覧は `quizeum-core` の `listActiveTags`（または同等 API）に依存する。タグチップ配列とキーワード・フィルタ条件の AND 合成検索は `searchQuizzes` に依存する。出題形式の解決ルールはコア／共有ユーティリティと一致させること。

## 要件

### 要件 1: ホーム画面 (Page: `/`)
**目的:** Quizeumのユーザーとして、タブ・ジャンル・複合検索でクイズを探索したい。それにより興味のあるクイズを見つけやすくなる。

#### 受け入れ基準
1. **ファーストビューの最適化**:
   - The Play Flow UI は、巨大なテキストバナーを廃止または縮小し、そのスペースをクイズコンテンツ表示に割り当てること。
   - The Play Flow UI は、検索バーを画面最上部の目立つ位置に優先して配置し、カテゴリーナビゲーションよりも視覚的優先度を高くすること。
2. **カテゴリー（ジャンル）表示の整理**:
   - The Play Flow UI は、`listActiveGenres` 由来のジャンル一覧について、1行に収まるスタイリッシュな「ピル」形式で横スクロール可能に表示し、縦方向の占有スペースを最小化すること。
   - The Play Flow UI は、主要なカテゴリー以外を「すべて見る」ボタン内に隠し、初期UIをクリーンに保つこと。
   - When ジャンルピルをクリックしたとき、the Play Flow UI は、ホーム内フィルタではなく `/genres/[genreId]` へ遷移すること（UI専用オプションはローカル定義可）。
3. **コンテンツ優先のレイアウト**:
   - The Play Flow UI は、画面の主要部分にグリッド形式で多数のクイズカードを表示し、1画面で発見できるクイズの数を最大化すること。
   - ホーム画面は「新着順」「人気順」「トレンド順」「タイムライン」（認証時のみ）のタブパネルを維持し、視認性の高いスタイルで切り替えをサポートすること。
4. **検索機能および UI/UX**:
   - **統合検索入力**: ジャンル、タグ、クイズ名、作者名を単一の検索バーで同時に検索可能とすること。
   - **検索バーの UI/UX**:
     - When ユーザーが検索バーを表示したとき、the Play Flow UI は、検索バーの左側に検索（虫眼鏡）アイコンを表示すること。
     - When ユーザーが文字を入力したとき、the Play Flow UI は、検索バーの右側にワンクリックで入力を全クリアできる消去（X）ボタンを表示すること。
     - When ユーザーが検索バーをフォーカスしたとき、the Play Flow UI は、ネオン調のグラデーション光彩のアウトラインをスムーズなトランジションで表示すること。
   - **クイックサーチチップ**:
     - When ユーザーが検索バーの直下に配置された人気のタグ（#ウミガメのスープ、#難問など）やおすすめジャンルのネオンバッジチップをクリックしたとき、the Play Flow UI は、そのテキストを検索バーに入力し、即座に統合検索を実行すること。
   - **スケルトンスクリーン**:
     - While 検索クエリの実行中（読み込み中）である間、the Play Flow UI は、クイズカードのプレースホルダーとなるスケルトンスクリーン（骨組みアニメーション）を表示すること。
   - **難易度・問題数・プレイ状況フィルタ**: 従来どおり、展開可能なフィルターパネル（SlidersHorizontal トグル）内で詳細指定ができること。ジャンル入力サジェストも含む。
   - **検索の実行**:
     - When 検索キーワードまたはフィルター条件が変更されたとき、the Play Flow UI は、デバウンス（300ms）を適用して `searchQuizzes` を呼び出し、条件を適用した結果をグリッドに反映すること。
5. **クイズカードの魅力向上**:
   - The Play Flow UI は、クイズカードにおいて、単一の固定電球アイコン表示を廃止し、クイズの内容を表現する固有のサムネイル画像を表示すること（サムネイルがない場合は適切なフォールバック表示を行う）。
   - The Play Flow UI は、各クイズカード内にタイトル、作成者、難易度、評価（スター数）などの情報を、整理された読みやすいレイアウトで表示すること。
   - The Play Flow UI は、各クイズカード内に明確な「プレイ」または「挑戦する」ボタンを設置し、ユーザーの直感的なインタラクションを誘導すること。
   - When 認証済みユーザーがカード内のブックマークボタンを押したとき、the Play Flow UI は、ブックマーク数を更新すること。未認証ゲストが押した場合は `/login` へ遷移すること。

### 要件 2: クイズ詳細画面 (Page: `/quiz/[id]`)
**目的:** Quizeumのユーザーとして、クイズのメタ情報・評価・ランキングを確認しプレイモードを選びたい。それによりプレイ前に内容を把握できる。

#### 受け入れ基準
1. クイズ詳細画面は、タイトル、説明、サムネイル、ジャンル、タグ、難易度（1〜10）、ブックマークトグルを表示すること。
2. クイズ詳細画面は、`reviewBadge` と `reviewScore` を表示すること（7日間の再評価リセット期間中はマスク表示）。
3. クイズ詳細画面は、「通常モード」「模擬試験モード」「フラッシュカードモード」の3モードを選べるプレイパネルを表示すること。
4. プレイパネルでプレイ開始を押したとき、選択モードのクイズプレイ画面（`/quiz/[id]/play`）へ遷移すること。
5. ログイン中ユーザーが当該クイズの作成者である場合、「クイズを編集」ボタンを表示すること。
6. 「クイズを編集」を押したとき、クイズ編集画面（`/quiz/[id]/edit`）へ遷移すること。

### 要件 3: クイズプレイ画面 (Page: `/quiz/[id]/play`)
**目的:** クイズプレイヤーとして、カウントダウン・ヒント・セッション保護付きで解答したい。それにより公平で信頼できるプレイ体験を得られる。

#### 受け入れ基準
1. 通常または模擬試験モードで設問ごとに制限時間がある場合、カウントダウンを表示し、0秒で不正解として自動提出すること。
2. 模擬試験モードでは設問ごとの制限ではなくクイズ全体の制限時間を適用し、前の設問へ戻って見直せること。
3. 「ヒントを表示」を押したとき、対応するヒント文を表示すること。
4. プレイ中、解答状態を `localStorage` に継続保存し、リロードやクラッシュから復元できること。
5. プレイ中にオフラインになった場合、オフラインで完了でき、完了時にオフライン結果画面を表示すること。

### 要件 4: 水平思考クイズ（ウミガメのスープ）プレイモード
**目的:** クイズプレイヤーとして、2カラムのAIチャットで水平思考クイズをプレイしたい。それにより質問しながら真相に辿り着ける。

#### 受け入れ基準
1. `lateral-thinking` 形式のとき、左にAIチャット、右にスクロール可能なQ&A履歴リストの2カラムレイアウトに切り替えること。
2. 未認証の場合、水平思考プレイ画面読み込み前に認証画面（`/login`）へ遷移すること。
3. 質問送信後AI応答待ちの間、AIチャット下部に灰色で「・・・AIが質問を分析中です」を表示すること。
4. 履歴と完全一致する質問を再送したとき、即時にキャッシュ回答を「📋 既存の回答」バッジ付きで返すこと。
5. 無料ユーザーが当該クイズの1日20回制限に達したとき、入力を無効化し「本日の残り質問数: 0/20」を表示すること。
6. 「真相を解き明かす」で真相要約を送信しAPIが合格判定したとき、クリア演出のうえ結果画面へ遷移すること。

### 要件 5: クイズ結果画面 (Page: `/quiz/[id]/result`)
**目的:** クイズプレイヤーとして、スコアと解説を確認し品質フィードバックを送りたい。それにより学習しコンテンツを評価できる。

#### 受け入れ基準
1. 結果画面は正解数／総問題数、経過時間、正誤一覧、各設問のマークダウン解説を表示すること。
2. 結果画面は👍/👎の二値投票と体感難易度（1〜10）投票UIを表示すること。
3. 「問題の間違い指摘」でカテゴリ（誤字、事実、別解）と詳細を入力できるフォームを開けること。
4. 「作家にお礼リアクションを送る」で作成者へ感謝通知を送信できること。
5. オフライン時はオフライン表示とし、投票・指摘・リアクションを無効化し、同期キューイングを通知すること。

### 要件 6: 弱点克服プレイ画面 (Page: `/quiz/review`)
**目的:** 認証ユーザーとして、過去に間違えた設問を復習したい。それにより弱点を克服できる。

#### 受け入れ基準
1. 復習開始前にジャンル選択（「オールジャンル」含む）パネルを表示すること。
2. フィルタに合致する間違い設問を取得し復習セッションを開始すること。
3. 復習完了時、間違いリストを更新し `users.totalFailedQuestionsCount` を減算すること（`quizeum-core` 連携）。

### 要件 7: その他の探索・総合リーダーボード画面
**目的:** Quizeumのユーザーとして、全体ランキングやタグ・ジャンルで探索したい。それにより競技的・高品質なコンテンツを発見できる。

#### 受け入れ基準
1. 総合リーダーボード画面（`/leaderboard`）は「総合ハイスコア」「月間プレイ数」「作家ランキング」のタブを表示すること（クイズ単位LBとは別仕様）。
2. タグ一覧（`/tags/[tagName]`）・ジャンル一覧（`/genres/[genreName]`）は該当クイズを人気順または新着順で一覧表示すること。
3. ブックマーク画面（`/bookmarks`）は、要件 11 に従いブックマークしたクイズ・リスト・設問を分類表示し、トグル操作で解除できること。

### 要件 8: クイズ編集認可保護 (Page: `/quiz/[id]/edit`)
**目的:** 認証ユーザーとして、編集画面へのアクセスを適切に制限したい。それにより作成者以外が編集できないようにする。

#### 受け入れ基準
1. 未認証ゲストが編集画面に直接アクセスした場合、認証画面（`/login`）へ遷移すること。
2. 認証済みだが作成者でないユーザー（`user?.id !== quiz.authorId`）が直接アクセスした場合、「アクセス権限なし」を表示し編集フォームを描画しないこと。

### 要件 9: クイズ単位リーダーボード表示（Phase 5）
**目的:** Quizeumのユーザーとして、初回プレイとリプレイのランキングを分けて確認したい。それにより初見と再挑戦の記録を誤解なく比較できる。

#### 受け入れ基準
1. クイズ詳細画面は、「初回プレイランキング（上位5名）」と「リプレイランキング（上位5名）」を**別セクションまたはタブ**で表示すること（総合リーダーボード画面とは別UI）。
2. 各ランキング表は、順位・ユーザー表示名・**正解数**・**合計解答時間（秒）**・達成日の列を含むこと。
3. 表示順は、正解数の多い順を第一キー、同点時は合計解答時間の短い順を第二キーとすること（`quizeum-core` の順位規則と一致）。
4. 各セクションに記録がない場合、「まだ記録がありません」等の空状態を表示すること。全問正解者のみ表示する文言や挙動は用いないこと。
5. 初回プレイデータは `leaderboardFirstPlay` を読み取ること。未移行クイズで当該フィールドが空のときのみ、旧 `leaderboard` フィールドを初回プレイ側のフォールバックとして表示してよい。
6. リプレイデータは `leaderboardReplay` のみを読み取ること。
7. E2Eおよびアクセシビリティ用に、クイズLB領域に `data-testid="quiz-leaderboard"` を付与し、初回側に `data-testid="highscore-leaderboard"`（または同等）、各行に `data-testid="leaderboard-entry"` を付与すること。
8. クイズ詳細画面は、リーダーボード配列の更新・並び替えロジックを実装してはならない（読み取りと表示のみ。更新は `quizeum-core`）。

### 要件 10: ジャンルマスタ駆動の探索UI（Phase 6）
**目的:** Quizeumのユーザーとして、運営が承認したジャンルマスタに基づいてホームやジャンル一覧を探索したい。それにより新設・マージ後のジャンルが画面に即時反映される。

#### 受け入れ基準
1. ホーム画面は、マウント時に `listActiveGenres`（または同等のコア API）で有効ジャンル一覧を取得し、表示名とアイコン画像 URL を用いたジャンルナビを描画すること。
2. ホーム画面のジャンルナビは、ソースとしてハードコード定数配列（例: `GENRES`）を**正本として用いてはならない**（「すべて」等の UI 専用オプションはローカル定義可）。
3. ジャンルアイコンをクリックしたとき、**常に** `/genres/[genreId]` へ遷移し、URL パラメータはマスタのジャンル ID（英小文字キー）を用いること（ホーム上でのインライン絞り込みにアイコンクリックを用いてはならない）。
4. ホームの複合検索パネルは、ジャンルサジェスト付きテキスト選択・難易度・問題数・プレイ状況の変更時に `searchQuizzes` を呼び出し（フィルタ変更トリガー、適切なデバウンス可）、すべての条件を AND で適用した結果をクイズグリッドに反映すること。プレイ状況は要件 1.3 と同規則とする。ジャンル・難易度・問題数・プレイ状況・キーワードがすべて未指定のときは従来のタブ別取得を維持してよい。
5. ジャンル別一覧画面（`/genres/[genreName]`）は、マスタから `displayName` と `iconImageUrl` を表示し、生の URL パラメータ文字列のみを見出しに用いないこと（マスタ未ヒット時は ID をフォールバック表示可）。
6. ジャンル別一覧画面は、「新着」「人気」「トレンド」のソートタブを提供し、選択に応じて `getQuizzesByGenre(genreId, limit, sort)` を呼び出すこと（C2 読み取りによりマージ済み旧ジャンルのクイズも含む）。
7. タグ別一覧画面（`/tags/[tagName]`）は、canonical 優先の `getQuizzesByTag` を用い、新着／人気／トレンドのソート切替に対応すること（既存 UI パターンと同型でよい）。
8. 弱点克服画面（`/quiz/review`）のジャンル選択は、ハードコード一覧の代わりに `listActiveGenres` 取得結果と「オールジャンル」を表示すること。
9. 有効ジャンルが 0 件のとき、ホームのジャンルナビは空状態または案内メッセージを表示し、クラッシュしないこと。
10. マスタ取得に失敗したとき、ホームおよびジャンル一覧はエラーまたは再試行可能な状態を表示し、サイレントにハードコード一覧へフォールバックしてはならないこと。

### 要件 11: 分類ブックマークと設問リストプレイ UI（Phase 8）
**目的:** 認証ユーザーとして、ブックマークしたクイズ・リスト・設問を種類ごとに管理し、設問リストを連続プレイしたい。それにより学習用のお気に入りと設問コレクションを活用できる。

#### 受け入れ基準
1. 認証済みユーザーがブックマーク一覧画面を開いたとき、プレイフロー UI は、「クイズ」「リスト」「設問」の3タブを表示し、選択中タブのみ該当カテゴリの一覧を表示すること。
2. 未認証ユーザーがブックマーク一覧画面にアクセスした場合、プレイフロー UI は、認証画面（`/login`）へ遷移すること。
3. 「クイズ」タブが選択されているとき、プレイフロー UI は、`getBookmarkFeed`（または `getBookmarkedQuizzes`）の結果をカード一覧で表示し、各カードからブックマーク解除トグルとクイズ詳細への遷移を提供すること。
4. 「リスト」タブが選択されているとき、プレイフロー UI は、ブックマークしたクイズリストをカード一覧で表示し、各カードからブックマーク解除トグルとリスト詳細（`/list/[id]`）への遷移を提供すること。
5. 「設問」タブが選択されているとき、プレイフロー UI は、ブックマークした設問を、問題文の抜粋・親クイズタイトル・ブックマーク日時の降順で一覧表示し、各カードからブックマーク解除トグルを提供すること。
6. ユーザーが設問タブの設問カードを選択したとき、プレイフロー UI は、当該設問の親クイズのプレイ画面（`/quiz/[parentQuizId]/play`）へ遷移し、当該設問から解答を開始できる導線を提供すること（単一設問プレイまたは当該設問インデックスへのスクロールでよい）。
7. 認証済みユーザーが通常または模擬試験プレイ中に設問を表示しているとき、プレイフロー UI は、当該設問のブックマーク登録／解除トグルを表示し、`toggleBookmark`（`targetType: 'question'`）を呼び出すこと。
8. 認証済みユーザーがクイズ結果画面で正誤一覧を表示しているとき、プレイフロー UI は、各設問行にブックマーク登録／解除トグルを表示し、親クイズが公開済みの設問のみ操作可能とすること。
9. 未認証ユーザーが設問ブックマーク操作を試みた場合、プレイフロー UI は、認証画面（`/login`）へ遷移すること。
10. ユーザーが `listType` が設問リストのリスト詳細画面（`/list/[id]`）を表示したとき、プレイフロー UI は、収録設問の順序付き一覧（問題文抜粋・親クイズタイトル）と「設問リストプレイ開始」ボタンを表示すること。
11. ユーザーが設問リストの「設問リストプレイ開始」を押したとき、プレイフロー UI は、リスト先頭の設問の親クイズプレイ画面へ遷移し、リスト内インデックスおよび `listId` をクエリまたはセッション状態で保持すること。
12. 設問リストプレイ中に1設問分の結果画面を完了したとき、プレイフロー UI は、リスト内に次の設問が残っている場合、次の設問のプレイ画面へ自動または明示ボタンで遷移し、最終設問完了後はリスト詳細または完了メッセージを表示すること。
13. ユーザーが `listType` がクイズリストのリスト詳細を表示したとき、プレイフロー UI は、従来どおり「リストプレイ開始」（クイズ単位連続プレイ）を提供し、設問リスト用 UI と混在させないこと。
14. プレイフロー UI は、`bookmarksCount` の更新や `attempts` の永続化ロジックを実装してはならない（`quizeum-core` のサービス呼び出しのみとする）。

### 要件 12: 探索検索のタグチップ化・サジェスト強化とクイズカード情報拡充（Phase 10）
**目的:** Quizeumのユーザーとして、ホームの検索バーでタグをチップ化しながらタグ名・ジャンル名のサジェストを受け、クイズ一覧カードから難易度・ジャンル・出題形式を一目で把握したい。それにより複数タグでの絞り込みが直感的になり、プレイ前の判断がしやすくなる。

#### 受け入れ基準

**検索バー — タグチップ**
1. ホーム画面の統合検索バーは、確定済みタグをチップ（バッジ）として検索入力の左側または内部に表示し、未確定の自由入力テキスト用フィールドを併設すること。
2. When ユーザーが検索入力中にスペースキーを押したとき、the Play Flow UI は、スペース直前の入力トークン（先頭の `#` は除去して正規化）をタグチップとして追加し、当該トークンを入力欄から除去すること。トークンが空または空白のみのときはチップを追加してはならない。
3. When ユーザーが検索入力中に Enter キーを押したとき、the Play Flow UI は、スペース確定と同一規則でタグチップを追加してよい（キーボード操作の代替確定として）。
4. When ユーザーが同一タグを重複してチップ化しようとしたとき、the Play Flow UI は、重複チップを追加してはならない。
5. When ユーザーがタグチップ上の削除（×）操作を行ったとき、the Play Flow UI は、当該チップを除去し、検索結果を再取得すること。
6. When ユーザーが消去（X）ボタンを押したとき、the Play Flow UI は、自由入力テキスト・すべてのタグチップ・検索バーに紐づくジャンル選択状態をクリアし、タブ別取得へ復帰してよいこと（フィルタパネル内の難易度・問題数・プレイ状況は従来どおり個別クリア可）。

**検索バー — タグ・ジャンルサジェスト**
7. While ユーザーが統合検索バーの自由入力欄に1文字以上入力している間、the Play Flow UI は、入力値に部分一致するタグ候補とジャンル候補をドロップダウン（またはリストボックス）で表示すること。ジャンル候補は表示名および ID の双方にマッチしてよい。
8. When ユーザーがサジェスト一覧からタグ候補を選択したとき、the Play Flow UI は、正規化済みタグ名のチップを追加し、自由入力欄をクリアすること。
9. When ユーザーがサジェスト一覧からジャンル候補を選択したとき、the Play Flow UI は、当該ジャンルを検索のジャンル条件として適用し（フィルタパネルのジャンル選択と同期してよい）、自由入力欄をクリアすること。
10. When タグマスタまたはジャンルマスタの取得に失敗したとき、the Play Flow UI は、サジェストを空またはエラー表示とし、サイレントにハードコード候補へフォールバックしてはならない。自由入力によるキーワード検索および既存フィルタは可能な範囲で継続してよい。

**検索バー — クイックサーチチップ連携**
11. When ユーザーが検索バー直下のクイックサーチチップ（例: `#ウミガメのスープ`）をクリックしたとき、the Play Flow UI は、プレーンテキストを入力欄へ流し込むのではなく、対応するタグチップを追加し、統合検索を即時実行すること。

**検索の実行 — チップとキーワードの合成**
12. When タグチップ・自由入力キーワード・フィルタパネル条件（ジャンル・難易度・問題数・プレイ状況）のいずれかが変更されたとき、the Play Flow UI は、デバウンス（300ms）後に `searchQuizzes` を呼び出し、すべての条件を AND で適用した結果をクイズグリッドに反映すること。
13. When 複数のタグチップが存在するとき、the Play Flow UI は、各タグ条件を AND（すべてのタグを含むクイズ）として検索すること。
14. When 自由入力キーワードが残っているとき、the Play Flow UI は、タイトル・説明・作者名等のテキスト部分一致検索にキーワードを用い、タグチップ条件と AND 合成すること。
15. When タグチップ・キーワード・ジャンル・難易度・問題数・プレイ状況がすべて未指定のとき、the Play Flow UI は、従来のタブ別取得（新着・人気・トレンド・タイムライン）を維持すること。

**クイズカード — 表示項目の拡充（ホーム・ジャンル一覧・タグ一覧共通）**
16. The Play Flow UI は、クイズ一覧で用いる共通クイズカードコンポーネントにおいて、難易度を **数値併記の星表記**（例: `★ 7`、難易度 1〜10 の整数値を星アイコンと併記）で表示すること。プログレスバー形式の難易度表示は用いてはならない。
17. The Play Flow UI は、各クイズカードにジャンル表示名を表示すること。ジャンルマスタから表示名を解決できるときはマスタの `displayName` を用い、未解決時はクイズに保存されたジャンル ID または名称をフォールバック表示してよい。
18. The Play Flow UI は、各クイズカードに出題形式の日本語ラベル（例: 選択式、記述式、ウミガメのスープ、複合形式 等）を表示すること。クイズに形式フィールドが無い旧データについては、設問構成から推定した形式ラベルを表示してよい（推定規則はコア／共有ユーティリティと一致させること）。
19. The Play Flow UI は、ホーム画面のクイズグリッド、ジャンル別一覧画面（`/genres/[genreName]`）、タグ別一覧画面（`/tags/[tagName]`）で、同一の共通クイズカードコンポーネントを用い、表示項目（難易度星表記・ジャンル・出題形式・評価・ブックマーク・プレイ導線）を揃えること。ジャンル／タグ一覧専用の重複インラインカード実装を正本として維持してはならない。
20. While クイズ一覧の検索またはソート結果の読み込み中である間、the Play Flow UI は、要件 1 のスケルトンスクリーンを表示し続けること。

**アクセシビリティ・テスト支援**
21. The Play Flow UI は、タグチップに削除可能であることを示すアクセシブル名を付与し、サジェスト一覧をキーボード（矢印・Enter）で操作可能にすること（既存ジャンルサジェストと同程度でよい）。
22. The Play Flow UI は、統合検索のタグチップ領域に `data-testid="search-tag-chips"`、各タグチップに `data-testid="search-tag-chip"`、サジェスト項目に `data-testid` プレフィックス（例: `search-suggest-tag-` / `search-suggest-genre-`）を付与すること。クイズカードの難易度表示に `data-testid="quiz-card-difficulty"`、ジャンルに `data-testid="quiz-card-genre"`、出題形式に `data-testid="quiz-card-format"` を付与すること。

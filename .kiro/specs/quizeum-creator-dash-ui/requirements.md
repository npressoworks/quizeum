# 要件定義書: quizeum-creator-dash-ui

## はじめに
本ドキュメントは、クイズ投稿SNS「quizeum」におけるクイズ作成・編集、作家ダッシュボード（アナリティクスおよび間違い指摘管理）、およびリストの作成・編集・エクスポートを含む、クリエイター（作家）向けフロントエンドUI要件を定義します。

**Phase 6（2026-06）**: クイズエディタのジャンルセレクトを `metadata_genres` マスタ（`listActiveGenres`）駆動に切り替え、ジャンル新設可決後の選択肢反映と play-flow 探索 UI との ID 一貫性を確保する。

**Phase 8（2026-06）**: リスト作成時の `listType`（クイズリスト／問題リスト）選択、問題リスト編集（問題検索・アタッチ・DnD・エクスポート）、および作問エディタでの過去自作クイズ検索・参照リンク追加 UI を追加する（永続化・検証は `quizeum-core` が担当）。

**Phase 12（2026-06）**: 作家ダッシュボード（`/creator/dashboard`）の Server Component 化、静的フレーム先行表示、および Suspense と Streaming による非同期スケルトン表示への移行を追加します。また、作問エディタの主要テキストエリア（説明文・問題文・真相・解説文）の自動伸長、過去自作クイズ検索の問題文・正解テキスト対応、参照リンク操作時の成功フィードバック、問題文ヒット時のアコーディオン自動展開、リンク済み問題の「リンク解除」、およびテストプレイ後の編集画面復帰時における問題数の不変保持を追加します（問題文・正解テキスト照合ロジックは `quizeum-core` が担当）。

**Phase 13（2026-06）**: 難易度の5段階化（1〜10から1〜5への変更）に伴い、作問エディタの難易度スライダー入力を1〜5の5段階へ変更します。

**Phase 20（2026-06-09）**: 〇×問題（`true-false`）を出題形式として選択可能にし、作問時は正解トグル（「〇が正解」「×が正解」）のみで設定できる UI を追加します。選択肢テキストの自由編集は提供しません（永続化・検証は `quizeum-core` が担当）。


## 境界コンテキスト
- **対象範囲（In scope）**:
  - クイズ新規作成・編集画面における動的問題管理、問題タイプ（選択肢/短答）の切り替え。
  - クイズタグ入力時の「自動名寄せ正規化」と類似 canonical タグ存在時のインラインサジェスト警告UI。
  - 公開申請時の Zod スキーマを用いた厳格な検証エラーリストインライン表示。
  - 作家ダッシュボードにおけるアナリティクスグラフ（累計プレイ数、評価、ブックマーク）およびクイズ個別問題解答割合グラフ。
  - プレイヤーからクローズドに送信された「間違い指摘フィードバック」の一覧表示と修正アクション動線。
  - 自分が作成したクイズデータのワンクリック一括 JSON エクスポート機能。
  - クイズリスト詳細画面における収録クイズ一覧と `listId` / `mode = 'list'` による連続プレイ開始 UI。
  - クイズリスト作成・編集画面におけるクイズ検索アタッチ、ドラッグ＆ドロップによる順序並べ替え、およびリストパッケージ JSON エクスポート。
  - **Phase 8**: リスト作成時の `listType` 選択、問題リスト向けの問題検索・アタッチ・並び替え・エクスポート UI、クイズエディタ内の過去自作クイズ検索パネルと問題の参照リンク追加 UI。
  - **Phase 12**: 作家ダッシュボード（`/creator/dashboard`）の Server Component 化、静的フレーム（ヘッダー、タイトル、各種アクションボタン等）の先行レンダリング、データ解決待ち中の部分スケルトン表示（`Suspense` 適用）。作問エディタの説明文・問題文・真相・解説文テキストエリアの自動伸長、過去自作クイズ検索パネルの問題文・正解テキスト検索、参照リンク成功メッセージ、問題文ヒット時のアコーディオン自動展開、リンク済み問題のリンク解除、テストプレイ後の編集画面ドラフト復元。
  - **Phase 20**: クイズ全体の出題形式として「〇×式」の選択、複合形式における問題タイプ「〇×」の切り替え、正解トグル（〇／×）のみの作問 UI。
- **対象外（Out of scope）**:
  - 外部クイズデータのインポート UI およびインポート処理本体（仕様変更によりインポート機能は完全に廃止されたため、UI 領域は一切設置しない）。
  - **Phase 8**: ブックマーク一覧の3タブ UI、プレイ／結果画面での問題ブックマーク、問題リスト連続プレイの遷移ロジック（`quizeum-play-flow-ui`）。プロフィールのリストタイプ別タブ表示（`quizeum-auth-profile-ui`）。
  - **Phase 12**: 作問エディタ以外の画面（リスト編集、プロフィール編集、管理画面等）のテキストエリア自動伸長の一括適用。過去自作クイズ検索のサーバー側全文検索インデックス新設。作家ダッシュボード以外の一部の管理画面等の非同期ローディング一括適用（主要画面以外は本フェーズの対象外）。
  - **Phase 20**: 〇×問題のプレイ回答 UI（1タップ 〇／× ボタン）（`quizeum-play-flow-ui`）。選択肢テキストの自由編集 UI。
- **隣接システムへの期待（Phase 6）**:
  - ジャンルマスタ読み取り・公開時 `canonicalGenreId` 解決は `quizeum-core` が担当。本スペックはエディタ UI の選択肢表示とリフレッシュのみ。
  - ジャンル新設申請・投票 UI は `quizeum-moderation-governance-ui`（`/community/genres`）が担当。
- **隣接システムへの期待（Phase 8）**:
  - リスト `listType` の永続化、問題追加の公開検証、`addQuestionToList` / `removeQuestionFromList`、`exportQuestionList`、`reorderQuestionList`（問題 ID）は `quizeum-core` が担当。
  - 過去自作クイズ検索は `searchAuthorQuizzes`、参照リンク保存は `QuizService.saveQuiz` の参照パス（Copy-on-Write 切り離し含む）に依存する。
  - 問題リストの「問題リストプレイ開始」ボタンと連続遷移は `quizeum-play-flow-ui`（要件 11）が担当。本スペックはリスト詳細の編集導線とメタ表示を担当する。
- **隣接システムへの期待（Phase 12）**:
  - 過去自作クイズ検索における問題文・正解テキストの照合ロジックは `quizeum-core`（`searchAuthorQuizzes` 拡張）が担当。本スペックは検索 UI・結果一覧・リンク操作フィードバックを担当する。
  - テストプレイの解答判定・結果保存・統計非記録は `src/lib/test-play.ts` およびテストプレイ画面が担当。本スペックはエディタ状態の `sessionStorage` 保存と復帰時の問題数保持を担当する。
  - ログイン必須画面に対する Next.js Middleware でのサーバーサイドリダイレクト（Cookie ベース認証）。

## 要件

### 要件 1: クイズ作成・編集画面 (Page: `/quiz/create`, `/quiz/[id]/edit`)
**目的:** クイズ作成者として、タグ正規化・サジェスト・厳格な検証付きのビジュアルエディタでクイズを下書き・編集・公開したい。それにより高品質なクイズを効率よく制作できる。

#### 受け入れ基準
1. クイズエディタ画面は、タイトル、説明、サムネイルアップロード、難易度（1〜5スライダー）、ジャンルセレクト、タグ入力（最大5個）のメタデータフィールドを表示すること。**Phase 6 以降**: ジャンルセレクトの選択肢は `listActiveGenres` 由来とする（要件 5 参照）。
2. クイズエディタ画面は、ジャンルセレクトの隣に「新しいジャンルを申請する」リンクを表示し、ジャンル新設申請画面へ遷移できること。
3. 作成者がタグを入力したとき、クイズエディタ画面は、テキストを正規化（トリム、小文字化、空白・記号除去）し、類似する canonical タグが存在する場合は「推奨: 類似するタグ #TagName が既に存在します。既存のタグを使用することをお勧めします。」のインライン警告を表示すること。
4. クイズエディタ画面は、問題の動的追加・削除、および問題タイプの「選択式 (multiple-choice)」／「短答文字入力式 (text-input)」切り替えを可能にすること。
5. 作成者がクイズの公開を試みたとき、クイズエディタ画面は、Zod スキーマ検証を実行し、統合制約違反（問題なし、正解未指定等）がある場合は明確な赤色エラーリストを表示すること。
6. クイズエディタ画面は、作成者が任意のタイミングで「下書き保存」をクリックし、検証エラーなく `QuizService.saveQuiz(status='draft')` 経由で下書きを Firestore に保存できること。

### 要件 2: 作家ダッシュボード (Page: `/creator/dashboard`)
**目的:** クイズ作成者として、アナリティクス・指摘対応・エクスポートを一元的に行いたい。それによりパフォーマンスを把握しクイズを維持できる。

#### 受け入れ基準
1. 作家ダッシュボード画面は、累計プレイ数・平均評価・ブックマーク数のアナリティクス概要を折れ線／棒グラフ等で表示すること。
2. 作家ダッシュボード画面は、各クイズの問題別解答割合（選択肢の円グラフ等）を詳細パネルで表示すること。
3. 作家ダッシュボード画面は、プレイヤーからのクローズド指摘（誤字、事実誤認、別解等）のキューを一覧表示すること。
4. 作成者が指摘カードの「修正する」をクリックしたとき、システムは、該当問題がプリロードされたクイズエディタ画面へ遷移すること。
5. 作家ダッシュボード画面は、「クイズ一括エクスポート」ボタンで全クイズ（下書き含む）を単一のダウンロード可能な JSON パッケージにまとめること。

### 要件 3: クイズリスト詳細画面 (Page: `/list/[id]`)
**目的:** Quizeum のユーザーとして、リストの内容を確認し連続プレイを開始したい。それにより整理されたクイズコレクションに取り組める。

#### 受け入れ基準
1. リスト詳細画面は、リストタイトル、説明、作成者アバター、カバー画像、およびクイズリストと問題リストを区別する種別表示（`listType` 未設定時はクイズリストとして表示）を表示すること。
2. リストがクイズリスト（`listType` が `quiz` または未設定）であるとき、リスト詳細画面は、収録クイズのカード一覧を表示すること。
3. リストが問題リスト（`listType` が `question`）であるとき、リスト詳細画面は、収録問題を順序付き一覧で表示し、問題文抜粋と親クイズタイトルを含めること（連続プレイ開始は `quizeum-play-flow-ui` 要件 11 が担当）。
4. ユーザーがクイズリストで「リストプレイ開始」をクリックしたとき、リスト詳細画面は、従来どおり `attempts.listId` と `mode: 'list'` によるクイズ単位の連続プレイを開始すること。
5. 認証済みユーザーがリスト作成者である場合、リスト詳細画面は、`listType` に関わらず「リストを編集する」ボタンを表示すること。

### 要件 4: リスト作成・編集画面 (Page: `/list/create`, `/list/[id]/edit`)
**目的:** クイズ作成者として、検索とドラッグ＆ドロップでリストを編集したい。それによりテーマ別のコンテンツコレクションをパッケージ化できる。

#### 受け入れ基準
1. リスト編集画面は、タイトル、説明、カバー画像、公開／非公開トグル、および `listType` に応じたコンテンツアタッチパネル（要件 6）を表示すること。
2. リストタイプが `quiz` のとき、リスト編集画面は、クイズ検索・アタッチ、ドラッグ＆ドロップ並び替え、クイズリスト JSON エクスポートを従来どおり提供すること。
3. リストタイプが `question` のとき、リスト編集画面は、要件 6 に従い問題のアタッチ・並び替え・エクスポートを提供すること。

### 要件 5: クイズエディタのジャンルマスタ連携（Phase 6）
**目的:** クイズ作成者として、承認済みジャンルマスタに沿ったジャンルセレクトを使いたい。それにより探索・モデレーションルールと一致したジャンルで公開できる。

#### 受け入れ基準
1. クイズエディタ画面は、マウント時に `listActiveGenres`（または同等のコア API）で有効ジャンル一覧を取得し、各 option にマスタの `displayName` と値 `genreId`（ドキュメント ID）を表示すること。
2. クイズエディタ画面は、選択可能ジャンルの正本としてハードコードされた `<option>` 一覧を用いてはならない。
3. クイズエディタ画面は、ジャンルセレクトの隣に「新しいジャンルを申請する」リンクを維持し、`/community/genres` へ遷移できること。
4. 作成者がジャンル新設フローから戻ったあとエディタがフォーカスを得たとき、または明示的な再試行が行われたとき、クイズエディタ画面は、有効ジャンル一覧を再取得し、新規承認ジャンルをフルリロードなしで選択肢に反映すること。
5. 保存済み下書きの `genre` が現行の有効一覧に含まれないとき（レガシーまたはマージ保留）、クイズエディタ画面は、現在値を表示したまま保存を許可すること（公開時検証は `quizeum-core` に委譲）。
6. ジャンルマスタ取得に失敗した場合、クイズエディタ画面は、エラーまたは再試行 UI を表示し、サイレントにハードコード一覧へフォールバックしてはならない。
7. クイズエディタ画面は、選択した `genreId` を `QuizService.saveQuiz` / 公開フローへ従来どおりの形状で送信すること（公開時の canonical 解決は core が担当）。

### 要件 6: 問題リストの作成・編集（Phase 8）(Pages: `/list/create`, `/list/[id]/edit`)
**目的:** クイズ作成者として、公開済み問題を複数ソースから集めて問題リストを作成・編集したい。それによりクイズ単位を超えた学習用コレクションを編集できる。

#### 受け入れ基準
1. 作成者がリスト編集画面で新規リストの作成を開始したとき、クリエイターダッシュボード UI は、保存前にリストタイプ `quiz` または `question` の選択を必須とし、`createQuizList` 経由で `listType` を永続化すること。
2. 既存リストを編集するとき、クリエイターダッシュボード UI は、現在の `listType` を表示し、作成後の `listType` 変更を許可してはならない。
3. リストタイプが `question` のとき、リスト編集画面は、クイズアタッチパネルを非表示にし、問題アタッチパネルを表示すること。
4. 作成者がアタッチする問題を検索したとき、リスト編集画面は、少なくとも次のソースでキーワード検索をサポートすること：自作公開クイズ内の問題、ブックマーク済み問題（`getBookmarkedQuestions`）、コアルールで追加可能な他者の公開問題。
5. 作成者が問題をアタッチ選択したとき、リスト編集画面は、`addQuestionToList` を呼び出し、問題文抜粋と親クイズタイトルをアタッチ一覧に表示すること。
6. 候補問題の親クイズが公開済みでない場合、作成者が当該問題のアタッチを試みたとき、クリエイターダッシュボード UI は、エラーを表示し、問題を追加してはならない。
7. 作成者がアタッチ済み問題を削除したとき、リスト編集画面は、`removeQuestionFromList` を呼び出し、フルページリロードなしで UI を更新すること。
8. 作成者がアタッチ済み問題の順序を変更したとき、リスト編集画面は、`reorderQuestionList`（または同等のコア API）を呼び出し、ドラッグ＆ドロップハンドルで順序を保持すること。
9. 作成者が問題リストのエクスポートを実行したとき、リスト編集画面は、`exportQuestionList` を呼び出し、コアのエクスポート規則に従ったリストメタデータと問題参照を含む JSON パッケージをダウンロードすること。
10. リストタイプが `quiz` のとき、リスト編集画面は、従来どおりのクイズ検索・アタッチ・並び替え・エクスポートを維持し、問題アタッチパネルを表示してはならない。

### 要件 7: 作問エディタの過去自作クイズ検索・参照リンク（Phase 8）(Pages: `/quiz/create`, `/quiz/[id]/edit`)
**目的:** クイズ作成者として、過去の自作クイズから問題を検索し、参照リンクで新規クイズに再利用したい。それにより再入力なしで効率的に作問できる。

#### 受け入れ基準
1. クイズエディタ画面は、認証済み作成者向けに折りたたみ可能なパネル「過去のクイズから問題を追加」（または同等のラベル）を表示すること。
2. 作成者がパネルでキーワードを入力またはタグを選択したとき、クイズエディタ画面は、現在の作成者にスコープした `searchAuthorQuizzes` を呼び出し、下書きを含む一致クイズを一覧表示すること。
3. 作成者が検索結果のクイズを展開したとき、クイズエディタ画面は、当該クイズの問題を問題文抜粋付きで表示し、個別問題のリンク選択を可能にすること。
4. 作成者が自作過去クイズ内の問題をリンク確認したとき、クイズエディタ画面は、エディタ状態に参照リンク（`linkKind: 'reference'`）として追加し、UI 層で問題内容を複製しないこと。
5. 作成者が所有しないクイズの問題をリンクしようとした場合、クリエイターダッシュボード UI は、当該操作を提供しない、または試行時に禁止エラーを表示すること。
6. クイズエディタ画面は、参照リンク済み問題と新規作成問題を視覚的に区別すること（例：バッジ「参照リンク」）。
7. 作成者が参照リンク済み問題の内容を編集したとき、クイズエディタ画面は、下書き保存または公開保存の前に、コアの Copy-on-Write 方針に従い保存時に切り離し用のローカルコピーが作成され得る旨を作成者に通知すること。
8. 作成者が下書きから参照リンク済み問題を削除したとき、クイズエディタ画面は、エディタ上からのみ参照を解除し、元クイズの問題レコードを削除してはならない。
9. クイズエディタ画面は、参照リンク済み問題 ID を `QuizService.saveQuiz` / 公開フローへ送信し、内容が未変更の場合は core が問題ドキュメントを複製せず参照として永続化できること。
10. クリエイターダッシュボード UI は、参照の永続化、所有権チェック、重複ドキュメント防止をクライアントで実装してはならない（いずれも `quizeum-core` の責務）。
11. When 作成者が過去自作クイズ検索パネルでキーワードを入力したとき、クイズエディタ画面は、クイズのタイトルおよび説明文に加え、各問題の問題文および問題タイプに応じた正解テキスト（選択式・正誤式の正解選択肢文、記述式・即答式・連想式の正解候補、並び替え式の各要素文、ウミガメのスープの必須正解キーワード）でも一致する自作クイズを検索結果に含めること。
12. The クイズエディタ画面は、過去自作クイズ検索パネルのキーワード入力欄に、タイトル・説明・問題文・正解テキストが検索対象であることを示すプレースホルダーまたは説明文を表示すること。
13. When 作成者が過去自作クイズ検索パネルで問題のリンク操作を成功したとき、クイズエディタ画面は、リンクが完了したことを作成者が認識できる成功メッセージを表示すること。
14. If 作成者が既にリンク済みの問題を検索パネルから操作するとき、クイズエディタ画面は、「リンク」ボタンの代わりに「リンク解除」ボタンを表示し、リンク解除によりエディタ上の当該問題のみを削除すること（要件 7.8）。重複「リンク」操作は防止すること。
15. When 作成者が過去自作クイズ検索パネルでキーワードを入力し、いずれかの問題の問題文がキーワードに一致したとき、クイズエディタ画面は、当該クイズのアコーディオンを自動展開し、一致する問題を即時表示すること。問題文ヒットの問題は正解テキストのみヒットの問題より先頭に表示してよい。
16. When 作成者がリンク済み問題の「リンク解除」を実行したとき、クイズエディタ画面は、リンク解除が完了したことを作成者が認識できるフィードバック（成功メッセージ等）を表示してよい。

### 要件 8: 作問エディタのテキストエリア自動伸長（Phase 12）(Pages: `/quiz/create`, `/quiz/[id]/edit`)
**目的:** クイズ作成者として、長文を入力するときにテキストエリアが内容に合わせて自動的に広がるようにしたい。それによりスクロールや手動リサイズなしで快適に作問できる。

#### 受け入れ基準
1. The クイズエディタ画面は、クイズ全体の説明文、各問題の問題文、ウミガメのスープ問題の真相入力、各問題の解説文のテキストエリアを表示すること。
2. When 作成者が上記テキストエリアに複数行のテキストを入力または削除したとき、クイズエディタ画面は、入力内容の行数に応じてテキストエリアの表示高さを自動的に拡張または縮小すること。
3. While テキストエリアが自動伸長しているとき、クイズエディタ画面は、作成者による手動リサイズも許可してよい。
4. The クイズエディタ画面は、タイトル・タグ・必須正解キーワード入力等の単一行用途フィールドを、本要件の自動伸長対象に含めてはならない。
5. When 作成者が既存の下書きまたは編集対象クイズを開いたとき、クイズエディタ画面は、保存済みの長文が含まれる上記テキストエリアについて、初回表示時点で内容に見合った高さを表示すること。

### 要件 9: テストプレイと編集画面復帰（Phase 12）(Pages: `/quiz/create`, `/quiz/[id]/edit`, `/quiz/test-play/*`)
**目的:** クイズ作成者として、テストプレイ後に編集途中のドラフトを失わず編集画面へ戻りたい。それにより本番保存前にプレイ検証と作問の継続を安全に行える。

#### 受け入れ基準
1. 作成者がクイズエディタからテストプレイを開始したとき、クイズエディタ画面は、未保存のエディタ状態（問題配列を含む）を `sessionStorage` に保存し、テストプレイ画面へ遷移すること。
2. 作成者がテストプレイ画面または結果画面から「編集画面に戻る」を選択したとき、システムは、元の編集パス（`/quiz/create` または `/quiz/[id]/edit`）へ `fromTestPlay=1` クエリ付きで遷移すること。
3. When 編集画面が `fromTestPlay=1` でマウントされたとき、クイズエディタ画面は、認証ユーザーと `sourcePath` が一致する `sessionStorage` のドラフトを1回だけ消費し、エディタ状態へ復元すること。
4. When テストプレイドラフトの復元に成功したとき、クイズエディタ画面は、復元した問題数を変更せず表示し、Firestore からの通常ロード（`getQuiz`）および新規作成時のデフォルト問題追加を実行してはならない。
5. When テストプレイドラフトの復元が完了したとき、クイズエディタ画面は、`fromTestPlay` クエリを除去した URL へ置換遷移し、同一セッション内の再マウントで問題が重複しないこと。
6. If `fromTestPlay=1` が付与されているが認証ユーザーが未確定のとき、クイズエディタ画面は、通常ロード（`getQuiz` / デフォルト問題追加）を開始してはならない。
7. If テストプレイ payload が欠落または期限切れのとき、クイズエディタ画面は、クエリを除去して復帰し、問題の重複追加を行ってはならない（編集モードでは以降の通常ロードに委ねてよい）。

### 要件 10: ダッシュボードの非同期表示最適化（Phase 12 追加）
**目的:** クイズ作成者として、作家ダッシュボードにアクセスした際、画面全体の白紙ローディングを待つことなく、静的なヘッダーや新規作成アクションが即座に表示され、データが揃った箇所から順番にダッシュボードの分析結果や指摘キューが表示されるようにしたい。これにより、待機時のストレスや画面の点滅による不快感を防ぎ、即時のアクション（クイズ作成等）を開始できる。

#### 受け入れ基準

**作家ダッシュボード画面における非同期表示最適化**
1. When [クイズ作成者が作家ダッシュボード（`/creator/dashboard`）にアクセスしたとき], the [Creator Dashboard UI] shall [サーバーコンポーネントとしてヘッダー枠、タイトル、および「クイズ新規作成」等のアクションボタンを含む静的なページフレームを即座にレンダリングし、Next.jsのStreaming機能を通じてクライアントへ送信すること]。
2. While [累計統計データ（プレイ数、評価、ブックマーク数等）がロード中である間], the [Creator Dashboard UI] shall [統計カードの表示エリアに、専用のスケルトンプレースホルダーを表示すること]。
3. When [累計統計データのロードが完了したとき], the [Creator Dashboard UI] shall [スケルトン表示領域を、実際の統計カードデータ（数値等）に差し替えること]。
4. While [作成クイズ一覧データがロード中である間], the [Creator Dashboard UI] shall [クイズ一覧表示エリアに、専用のローディングスケルトンを表示すること]。
5. When [作成クイズ一覧データのロードが完了したとき], the [Creator Dashboard UI] shall [スケルトン表示領域を、実際のクイズ行リストに差し替えること]。
6. While [間違い指摘フィードバックキューがロード中である間], the [Creator Dashboard UI] shall [指摘一覧表示エリアに、専用のローディングスケルトンを表示すること]。
7. When [間違い指摘フィードバックキューのロードが完了したとき], the [Creator Dashboard UI] shall [スケルトン表示領域を、実際の指摘カード一覧に差し替えること]。
8. While [アナリティクスグラフデータ（プレイ数、評価推移）がロード中である間], the [Creator Dashboard UI] shall [グラフ表示エリアに、専用のグラフ用スケルトンを表示すること]。
9. When [アナリティクスグラフデータのロードが完了したとき], the [Creator Dashboard UI] shall [スケルトン表示領域を、実際のチャート（折れ線・円グラフ等）に差し替えること]。

**アクセシビリティ・テスト支援**
10. The [Creator Dashboard UI] shall [統計カードのスケルトン領域に `data-testid="stats-skeleton"`、クイズ一覧のスケルトン領域に `data-testid="quiz-list-skeleton"` を付与すること]。
11. The [Creator Dashboard UI] shall [指摘キューのスケルトン領域に `data-testid="feedback-list-skeleton"`、グラフのスケルトン領域に `data-testid="charts-skeleton"` を付与すること]。

**クイズ作成・編集画面、リスト関連画面の非同期表示最適化**
12. When [ユーザーがクイズ作成画面（`/quiz/create`）または編集画面（`/quiz/[id]/edit`）にアクセスしたとき], the [Creator Dashboard UI] shall [サーバーコンポーネントとして戻るボタン、入力フォームの静的フレーム（見出し、保存・公開等のアクションエリア）を即座にレンダリングし、Next.jsのStreaming機能を通じてクライアントへ送信すること]。
13. While [編集対象のクイズデータやジャンルマスタ、タグリストの取得中である間], the [Creator Dashboard UI] shall [フォーム内部の各エリア（メタデータ入力、問題アコーディオン等）に専用のスケルトンプレースホルダーを表示すること]。
14. When [データのロードが完了したとき], the [Creator Dashboard UI] shall [スケルトン表示領域を、実際の編集フォームコンテンツに差し替えること]。
15. When [ユーザーがリスト詳細画面（`/list/[id]`）またはリスト作成・編集画面（`/list/create`, `/list/[id]/edit`）にアクセスしたとき], the [Creator Dashboard UI] shall [サーバーコンポーネントとして戻るボタン、コンテナ等の静的フレームを即座にレンダリングし、クライアントへ送信すること]。
16. While [該当リストデータやアタッチ対象のクイズ・問題データがロード中である間], the [Creator Dashboard UI] shall [リストコンテンツ表示領域に専用のスケルトンを表示すること]。

**アクセシビリティ・テスト支援（追加分）**
17. The [Creator Dashboard UI] shall [クイズエディタのスケルトン領域に `data-testid="quiz-editor-skeleton"`、リスト詳細・編集のスケルトン領域に `data-testid="list-editor-skeleton"` を付与すること]。

### 要件 11: 〇×問題の作問 UI（Phase 20）(Pages: `/quiz/create`, `/quiz/[id]/edit`)
**目的:** クイズ作成者として、〇×問題を他の出題形式と同様に選び、正解をトグルだけで設定したい。それにより素早く正誤問題を作れる。

#### 受け入れ基準

**クイズ全体の出題形式**
1. The [Creator Dashboard UI] shall [クイズ全体の出題形式選択に「〇×式」（`true-false`）を、選択式・記述式等と並ぶ選択肢として表示すること]。
2. When [作成者がクイズ全体の出題形式として「〇×式」を選択したとき], the [Creator Dashboard UI] shall [新規追加される全問題を `true-false` タイプとして作成し、問題ごとの形式切り替え UI を非表示（または固定表示）とすること]。
3. When [作成者が「〇×式」以外の出題形式から「〇×式」へ変更したとき], the [Creator Dashboard UI] shall [既存問題を `true-false` 形式へ一括変換する旨を確認し、同意後に各問題へ固定 〇／× 選択肢と正解トグル用の初期状態を設定すること]。

**複合形式における問題タイプ**
4. Where [クイズ全体の出題形式が「複合」（`mixed`）である場合], the [Creator Dashboard UI] shall [各問題のタイプ切り替えに「〇×」を、選択式・記述式・並び替えと並ぶ選択肢として表示すること]。
5. When [作成者が複合形式で問題タイプを「〇×」に切り替えたとき], the [Creator Dashboard UI] shall [当該問題を `true-false` タイプに変更し、選択式用の複数選択肢 UI を非表示にすること]。

**正解トグル（選択肢文言は固定）**
6. Where [問題タイプが `true-false` である場合], the [Creator Dashboard UI] shall [「正解はどちらですか？」等のラベルと、正解を「〇」または「✕」のいずれか一方に指定するトグル（または同等の2択 UI）のみを表示すること]。
7. The [Creator Dashboard UI] shall [`true-false` 問題について、選択肢テキスト「〇」「✕」を作成者が自由編集する入力欄を表示してはならない]。
8. When [作成者が正解トグルを変更したとき], the [Creator Dashboard UI] shall [内部状態上、固定ラベル「〇」「✕」の2選択肢のうち正解側のみ `isCorrect: true` となるよう反映すること]。
9. When [作成者が `true-false` 問題を下書き保存または公開保存したとき], the [Creator Dashboard UI] shall [`quizeum-core` の検証規則に従い、選択肢2件・正解1件のデータを送信すること]。

**参照リンク・形式変換**
10. When [参照リンク済みの `true-false` 問題を表示するとき], the [Creator Dashboard UI] shall [正解トグルを読み取り専用または切り離しポリシーに従い、要件 7 の参照問題ルールと整合させること]。
11. When [作成者が「複合」形式へ変更したとき], the [Creator Dashboard UI] shall [既存の `true-false` 問題は `true-false` タイプを維持し、複合形式で許可される他タイプへの自動変換を行わないこと]。

**境界・隣接**
12. The [Creator Dashboard UI] shall [〇×問題の正誤判定・`Quiz.format` 永続化・公開時の選択肢正規化をクライアントで実装してはならない（`quizeum-core` が担当）]。
13. The [Creator Dashboard UI] shall [プレイ時の 〇／× 1タップ回答 UI を本要件の範囲に含めない（`quizeum-play-flow-ui` が担当）]。

**アクセシビリティ・テスト支援**
14. The [Creator Dashboard UI] shall [正解トグル領域に `data-testid="true-false-correct-toggle"`、複合形式の問題タイプ「〇×」ボタンに `data-testid="question-type-true-false"` を付与すること]。
